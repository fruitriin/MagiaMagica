## モジュール設計指針 — *A Philosophy of Software Design* (Ousterhout, POSD)

オーナーが普段書いているコードの感覚がそのまま言語化されている本。**オーナーの設計判断の軸は POSD に沿っている**と考えて差し支えない。Claude が実装するとき、レビューするとき、設計を選ぶとき、迷ったら POSD の原則をそのまま判断軸に使ってよい (「これ POSD 的にどっち？」が成立する関係)。

- **深いモジュール (Deep Modules)**: インターフェースは小さく、内部の機能は大きく。`pub` を増やすほど浅いモジュールになる。新しい型や関数を `pub` にする前に「この境界はパススルー関数になっていないか」を確認する
- **情報隠蔽 (Information Hiding)**: 実装詳細は呼び出し側に漏らさない。フィールドを `pub` にする前に「これは設計の本質か、たまたま今そう書かれているだけか」を確認する。後者なら `pub(crate)` 以下に下げるか、コンストラクタを介させる
- **複雑性を下に押し下げる (Pull Complexity Downwards)**: 利用側が if 連発するくらいなら、提供側が複雑さを吸収する。Result の扱いや初期化の煩雑さは下位レイヤーで畳む
- **一般的なインターフェース (General-Purpose Interfaces)**: 1呼び出し1ユースケースの special-purpose API より、わずかに抽象度を上げた general-purpose API が長持ちする。ただし「YAGNI と相反する」と感じたら**そのときの自分の感覚**を優先してよい (Ousterhout 自身も Strategic vs Tactical の中で議論している)
- **エラーを存在しないものとして定義 (Define Errors Out of Existence)**: 「ありえない状態」は型で潰す。`Option<u32>` vs `u32 = 0` センチネル の選択は前者。`Result` で報告する前に、そもそもエラーが起きえないようにできないか考える。**`u32::try_from(..).unwrap_or(u32::MAX)` のようなセンチネルへの黙落ちは禁止** — 実用上起こらない超過は `expect("〜を超えることはない")` で明示的に落とす (Phase 1.3〜3.5 で4回再発したプロジェクト横断規約)
- **戦略的プログラミング (Strategic vs Tactical)**: いま動かすためだけの戦術コードを書くと負債が早期に積もる。Phase の境界では戦略的に整え、内部の試行錯誤では戦術的に許す
- **コメントは「コードからは読み取れないこと」を書く**: WHY、不変条件、設計上の制約、外部仕様への参照。コードを言い換えるコメントは書かない。spec の節番号 (`spec §4.2`) や Phase 1.x の根拠を埋め込むのは推奨
- **コメントを先に書く**: 設計判断が固まらないまま実装に入りそうなときは、ドキュメントコメントから書き始めると inference が早く回収できる

### POSD 観点で **やらないこと**

- 過剰な抽象化のための trait 切り出し (1実装しか想定がないなら struct のままにする)
- 早すぎる generic 化 (具体型で書いて、2 回目に同じ形が出てから抽出する)
- 「将来使うかも」だけが根拠のフィールド (将来必要なら破壊的に追加する。下記の方針)

## 後方互換性ポリシー (v1.0 未到達)

**メジャーバージョン (v1.0) に到達するまで、より良い設計を思いついたら積極的に破壊的変更を入れる**。

- 「既存テストが壊れるから」「JSON フォーマットが変わるから」を理由に劣った設計を温存しない
- 旧 API を残して新 API を増やすより、旧 API を消して新 API だけにする方を選ぶ (deprecate よりも remove)
- 例外: 既に外部に公開済みのフィールド・型・コマンドラインに対して、移行計画なしに突然壊すのはしない。Phase の境界 (Phase 1 → Phase 2) で破壊的変更をまとめる
- `Cargo.toml` の `version` は v1.0 までは自由に振る。`0.1.0` → `0.2.0` で破壊的変更を入れることを躊躇しない

これは Phase 1.1 で IR スケルトンを「将来 Phase 分のフィールドを最初から確保」した方針と矛盾しない。**スキーマの場所を確保するのは「今の最善の予想」**であり、後でより良い予想が立ったら遠慮なく入れ替える。`#[serde(default)]` は破壊的変更を妨げる装置ではなく、軽微な追加変更を低コストにする装置として位置づける。

### 判断のチェックリスト

新機能や設計変更で迷ったとき:

1. これは POSD の「深いモジュール」を浅くしていないか？
2. これは「複雑性の上方押し付け」になっていないか？ (呼び出し側が if 連発する形になっていないか)
3. 既存設計が劣っていることに気づいた → **v1.0 前なら直す**。コミットログに `[break]` プレフィックスを付ける
4. 「将来のため」だけが根拠 → 入れない

## ビルド・Lint・テスト

Phase 1.0 で cargo workspace を立ち上げた。以後のマイルストーンはこの土台に積み増す。
Phase 4.0.5 でフロントエンド (`web/`, Vue 3 + Vite+) が加わった。**ランタイム・パッケージマネージャは Bun** (Node.js / pnpm は不採用、オーナー指示 2026-06-11)。

```bash
# ビルド (magia-cli の build.rs が web/dist を bun で自動ビルドして rust-embed で同梱する)
cargo build --workspace

# Lint (clippy + rustfmt)
cargo clippy --workspace --all-targets -- -D warnings
cargo fmt --check

# テスト
cargo test --workspace

# フロントエンド検証 (fmt + lint + 型チェックの一括窓口) とビルド
cd web && bun run vp check && bun run build
```

ADDF フレームワーク自体のテストランナーも利用可能:

```bash
bash .claude/tests/run-all.sh
```

品質ゲートの Stage 1 では、上記の cargo コマンド一式に加えて `bash .claude/tests/run-all.sh` と `web/` の `vp check` + `bun run build` を実行する。

### バイナリサイズの方針 (オーナー確定 2026-06-11)

**バイナリサイズの肥大化は、それが意味のある機能 (syntect のハイライト、SPA 同梱など) の対価なら気にしない**。サイズを理由に機能や依存を削る判断はしない。`[profile.release]` の strip/lto のような「タダで縮む」最適化だけ維持する。

### 開発フロー (フロントエンド)

- **開発時は二段構成**: `magia serve <FILE>` (API、ポート 4747) + `cd web && bun run dev` (HMR、ポート 5173)。vite の proxy が `/state /spell /events` を 4747 へ転送するので、ブラウザは 5173 だけ見る。1コマンド起動は `scripts/dev-web.sh <FILE>`
- **本番配信は rust-embed**: `cargo build` 後の `magia serve` は同梱済み SPA を配信する (HMR なし)。フロント変更を本番経路で確認するときは再ビルドする
- Bun がない環境では cargo build が手順つきのエラーで止まる (https://bun.sh からインストール)

## ノウハウ記録の3観点

タスク実行中、以下の3つのタイミングで `/addf-knowhow` を使い知見を記録する。
ProgressTemplate の運用ルールに組み込み済み。

1. **コーディング知見**（実装フェーズ完了時）: 再利用可能なパターン、落とし穴、技術的判断とその根拠
2. **品質ゲート知見**（品質検証完了時）: レビューエージェントが検出した再発しうるパターン
3. **タスク総括**（完了処理時）: 計画と実装のギャップ、想定外だった点、次回同種タスクへの教訓
