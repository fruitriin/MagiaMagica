# Phase 4.1 — ピン中心ビュー (フォーカス + 距離フェード)

## 出典

- オーナー要望 (2026-06-11): 「中心関数をピンで固定、ピン直しで再レイアウト。フォーカスから遠い関数ほど小さく薄くして存在感を希薄化させる」
- 親計画: Phase 4.0 (ソース連動ペアビュー) の上に「フォーカス概念」を載せる

## 目的

dev-server を「1関数だけ大写し」から「**フォーカス関数を中心に据え、周辺関数を距離に応じて小さく薄く配置する太陽系的ビュー**」へ拡張する。
ピン操作で焦点を打ち直すと全体が再レイアウトされ、「どの関数を主役に据えて世界を見ているか」がインタラクションで切替わる。

RTS の自由ズームではなく、**焦点固定 + 距離フェード**モデルを採る (オーナー指定)。
連続ズームは扱わない。代わりに「ピンの打ち直し」が世界の再構築をもたらす。

## スコープ

### やること

- **フォーカス/周辺概念の導入**:
  - 任意の1関数を `focus` として URL `?pin=<name>` に記録
  - 周辺関数 = Phase 4.2 で定義する `ProximityIndex` から引いた近接関数群 (本計画では 4.2 のスタブ実装をまず使い、4.2 完了時に差し替え)
- **同心リング配置**:
  - 中央: フォーカス関数の魔法陣 (現在 serve が描画しているサイズ)
  - 内リング (距離=同 impl): 半径 r1, scale 0.55, opacity 0.85
  - 中リング (距離=同ファイル): 半径 r2, scale 0.35, opacity 0.6
  - 外リング (距離=呼び出し関係): 半径 r3, scale 0.25, opacity 0.4
  - 各リング上の関数は決定論的配置 (名前ソート → 等角度割付) で再描画ごとに位置が動かない
- **ピン操作**:
  - 周辺の関数シグネチャ盾 (縮小魔法陣 + 関数名チップ) をクリック → ピンが新関数に移動
  - キーボード: F = フォーカス中心、Tab = 次の周辺関数にカーソル、Enter でピン打ち直し
  - history.pushState で `?pin=<name>` を更新し、ブラウザ戻る/進むでピン遷移を辿れる
- **ピン遷移アニメーション**:
  - 旧フォーカスが中リング位置へ縮みつつ移動、新フォーカスが中心へズームイン
  - CSS transition 250〜400ms。`prefers-reduced-motion: reduce` 尊重で即時切替フォールバック
- **コードビュー側 (Phase 4.0 と統合)**:
  - 左ペインのソース表示は **フォーカス関数のみ**
  - 関数目次の代わりに「周辺関数のシグネチャテキスト (`fn name(args) -> ret`)」を距離順に列挙
  - 各エントリは関数名 + 引数を1行で。クリックでピン打ち直し
- 解析エラー時は周辺再計算をスキップし、直前のレイアウトを保持

### やらないこと (別計画)

- 近接度モデルの実装本体 (Phase 4.2)。本計画ではスタブ ProximityIndex (同 impl + 同ファイル順) で動かす
- 静止画レンダの拡張 (Phase 4.3)
- 呼び出し先ジャンプ (Phase 4.4) — 周辺リストには呼び出し先候補を出すが、魔法陣内記号をクリックしての遷移は別計画
- ワークスペース横断 (Phase 4.5) — 同ファイル外は本計画では距離∞扱い
- ベルカ式との併用 (Phase 3.5 の `?style=` と直交。両軸並存は将来課題)

## 設計上の判断

### 距離は「リング」で離散化する (連続ズームしない)

オーナー指定の「ピン中心固定、再レイアウトはピン打ち直し時のみ」を物理に翻訳:
- 連続ズームを許すと「中間状態でレイアウトが揺れる」誘惑が生まれ、決定論性が崩れる
- 距離をリング 3 段階に固定すれば、再レイアウトはピン打ち直しの一発で済み、ファイル監視更新時も差分配信が単純
- ズームインしないと記号は判読できない → 「リング外に出した関数は概観のみ」が成立し、情報の階層が明示される

### 配置は同心円 + 名前ソートの決定論

力学レイアウトを避ける:
- ファイル監視で関数が追加削除されたとき、力学だと既存関数の位置まで動いて目が泳ぐ
- 名前ソート → 角度等分なら、追加で他の関数の角度がわずかに動くだけ
- これは Phase 1.5 の決定論レイアウト方針 (knowhow `svg-deterministic-rendering.md`) と一貫

### 周辺関数の「描画コスト」を抑える

リング内の関数を全部フル魔法陣でレンダすると重い。以下で軽量化:
- 内リング: ring 構造のみ (補助リング・記号は描く)
- 中リング: 外周リング + シグネチャテキスト
- 外リング: 名前盾のみ (魔法陣の影 = 円 + 関数名)
- これを serve サーバ側で「縮小版 IR」として一度だけ生成しキャッシュ

### サーバ側でレイアウト計算する

クライアントでレイアウトすると IR を送る必要があり、SSE 更新時のペイロードが重くなる。
サーバが「現在のピンを中心としたレイアウト済み SVG (周辺含む)」を1枚返す方が単純。
`/spell/<focus>?with=neighbors` のような形にして、neighbors なしも残す (静的レンダ・テスト用)。

### `?pin=<name>` と Phase 4.0 の `?fn=<name>` の関係

`?fn` を `?pin` にリネーム。Phase 4.0 の段階では「ピン = 表示中の関数」と意味が一致するため、用語を統一する方が一貫性が良い。
Phase 4.0 で実装した `?fn` パーサは互換目的で `?pin` を読むようにリネーム + フォールバックは置かない (v1.0 前破壊的変更ポリシー)。

## 受け入れ基準

- [ ] `magia serve <FILE>` の dev-server がフォーカス + 周辺リング構造で表示される
- [ ] 周辺関数の盾をクリックすると 250〜400ms のアニメーションで中心が移動する
- [ ] `prefers-reduced-motion: reduce` 環境ではアニメ無しで即切替
- [ ] `?pin=<name>` で初期フォーカスを指定できる。未知名は先頭関数 + status 表示
- [ ] ブラウザ戻る/進むでピン遷移履歴を辿れる
- [ ] 関数の追加削除で周辺リング配置が破綻しない (golden: 関数並びが名前ソートで決定論)
- [ ] コードビューにはフォーカス関数のみのソースが表示される
- [ ] `cargo test --workspace` / clippy / fmt 通過

## 実装ステップ

**実装フェーズ**

1. **magia-core**: スタブ `ProximityIndex` (同 impl / 同ファイル順、呼び出し関係は空)
2. **magia-core**: `render_focus_layout(focus: &FunctionEntry, index: &ProximityIndex) -> Svg` — 中央 + 3リング配置のレイアウト関数
3. **magia-core**: 周辺関数の「縮小表現」3段階 (`MiniRingLevel`) の SVG 生成
4. **magia-cli/serve**: `/spell/<focus>?with=neighbors` エンドポイント追加。SSE 更新時もこの経路で配信
5. **magia-cli/serve**: `?fn` → `?pin` リネーム + 404 ハンドリング更新
6. **フロント**: 周辺盾の DOM 構造 + クリックイベント + ピン遷移アニメ (CSS transition + class swap)
7. **フロント**: history.pushState + popstate でピン遷移履歴
8. **フロント**: キーボードナビ (F / Tab / Enter)
9. **エラー**: パース失敗時の周辺レイアウトスキップ
10. **テスト**: スタブ ProximityIndex の golden / レイアウトの決定論 / `?pin=` の URL 同期 / ピン遷移後の DOM 状態 / 未知 pin 名のフォールバック
11. **目視素材**: 既存 fixtures (medium_render_doc / write_document / dense_dispatch) でピン中心ビューを撮ってオーナー送付

## 想定リスク

- **盾上のテキスト可読性**: scale 0.25 の外リングで関数名が読めない可能性。実装時に外リングは「ホバーで関数名を tooltip 表示」フォールバックを用意
- **SVG ペイロード肥大**: 周辺関数 N 個の縮小版を毎回送ると重い。N が増えたら gzip or 縮小版だけ別エンドポイント化を検討 (本計画では fixture 規模で問題ない範囲)
- **`prefers-reduced-motion`の取り違え**: テスト時に motion 有無を両方確認
- **ProximityIndex スタブと本実装の境界**: 4.2 完了時の差し替えが破壊にならないよう、`Proximity` トレイトで抽象化しておく
