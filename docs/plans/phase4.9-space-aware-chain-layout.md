# Phase 4.9 — 空きスペースを使う鎖・補助陣レイアウト

## 出典

- オーナー判定 (2026-06-13 — 4.8 M2 後): 「雰囲気はいい感じ。`span_excerpt` の例なんかは
  陣が重なり合っちゃうシーンがあるから、召喚陣同士の距離を等間隔に固定しないで空き
  スペースに応じて延長するとか、空きスペースに配置できるように通常の召喚陣でも基準角で
  折り曲げるとかそういう工夫がいりそう」

## 目的

4.8 M1/M2 で得た「鎖 (Chain edge) + クロージャ補助陣 (AuxRingKind::Closure)」の構造は
そのままに、**配置だけ空きスペースを使う**形に進化させる。`span_excerpt` 級の混み合った
関数で補助陣どうし・鎖どうしの重なりを解消する。

## 現状の素朴さ (4.8 M1/M2 で意図的に残した形)

- **鎖**: `place_chain_glyphs` が**先頭の外向き単位ベクトル**に `CHAIN_STEP=40` 等間隔で
  直線配置する。途中の glyph が混んだ領域に突っ込んでも避けない
- **クロージャ陣**: 先頭 glyph は外向き / 鎖の途中の glyph は直交 (4.8 M2 後追加)。
  AUX_RING_RADIUS=48 のリングが固定距離で1つだけ — 連鎖したクロージャ陣の入れ子で
  「鎖→陣→鎖→陣」の波が同じ方向に重なる
- 衝突回避は鎖・glyph 係留リング双方とも未実施 (M1 から継続の宿題と明記)

## やる方向 (オーナー観察の言語化)

1. **鎖の延長 (等間隔 → 空きに応じて伸縮)**: 既存 fan glyph・補助リング・他の鎖と
   重なる前に CHAIN_STEP を伸ばす / 距離を取る。最短は今のまま、混んだら長くする
2. **基準角で折り曲げ**: 直進だけでは収まらない鎖は途中で角度を変える (S 字や L 字)。
   候補方向は外向き起点に ±π/4 などの離散値から「空いてる側」を選ぶ
3. **クロージャ陣の場所探し**: 親 glyph からの距離・角度を多候補生成し、占有マップ
   (既存配置を半径含めて畳んだもの) との衝突が無い最初の候補を採る
4. **占有マップ自体は共通基盤**: 鎖・補助陣・通常召喚印 (fan) の3経路から参照される

## やらない方向

- 鎖と補助陣を「物理シミュレーション」で動かす (決定論を壊す)
- 既存 fan の交差最小化結果の上書き (4.0.6 後半・1.8 で得た意匠を守る)
- 多モジュール対応 (4.5 と直交)

## 設計案 (着手時に精緻化)

### 占有マップ

- 配置済み Sigil の `(中心, 半径 + マージン)` を `BTreeMap<SigilId, Disc>` に集める
- 衝突判定は素朴な O(N²) で良い (ローカル関数1個のスケール — 数十〜数百程度)
- BTreeMap 走査 = 決定論的 (Phase 1.5 から続く規約)

### 鎖の延長 + 折り曲げ

```
方針: cursor から「外向き+候補角」×「距離倍率 1.0, 1.5, 2.0」の格子で
空きを探す。最初に衝突しない候補を採用。何もない場合は最遠候補
(最後尾) を採って警告フラグ (overflow=true) を IR に乗せる
```

候補角の最大ねじれを `CHAIN_TURN_MAX_RAD` (仮 π/6) で制限すると「曲がりすぎ」を抑制。

### クロージャ陣の場所探し

- 基準角 (鎖の途中は直交、先頭は外向き) ± `CLOSURE_SIBLING_SPREAD_RAD` を中心に
  「外向き 1.0 / 1.5 / 2.0 倍距離」× 「ねじれ -π/4, 0, +π/4」の候補から探索
- 兄弟扇形は基準角の段階で展開済み (4.8 M2 W1 対応) なので、空き探索は1陣ずつ独立

### M1 / M2 の分割

- **M1**: 占有マップ + 鎖の延長 (折り曲げなし、直線の伸ばすだけ)。意匠変化を最小に
- **M2**: 鎖の折り曲げ + クロージャ陣の場所探し。判定で M1 の感触を見てから設計確定

## 判定ポイント (オーナー目視)

- `span_excerpt` の混み具合がどこまで解消されるか (M1 / M2 別々の素材で送付)
- 鎖の折り曲げが「自然な迂回」に見えるか「不自然なジグザグ」に見えるか
- 等間隔から伸ばす方針の「最小は今と同じ」が保たれているか (短い鎖の意匠は不変)

## 想定リスク

- **決定論**: 多候補探索は順序依存になりやすい。候補生成順は固定 (距離倍率 → 角度 →
  符号) + 衝突判定の入力を BTreeMap で取る (規約踏襲)
- **意匠の連続性**: ファイル保存ごとに 1 行変えるだけで配置が大きく動くと「会話を切る」。
  候補順を「現状に最も近い」優先にして揺れを抑える
- **オーバーラン**: 全候補が衝突する場合の最後の手段 (最遠候補 + 警告フラグ) は M1 で
  確定させ、M2 でも踏襲する

## 実装結果メモ — M1 (2026-06-13)

### 実装内容

- **占有マップは新設せず `placed` (配置済み Sigil の中心 + 半径) をそのまま使う**
  (POSD: 新規構造体を増やさない最小実装)
- **鎖の延長**: place_chain_glyphs で後続候補位置を CHAIN_STEP * step を基点に、
  衝突があれば CHAIN_STEP/4 ずつ伸ばす離散グリッド。**上限は step 倍率**
  (CHAIN_STEP_MAX_FACTOR=4 倍まで — レビュー W1 で「全 step 共通の絶対上限」から修正)
- **意匠連続性**: 衝突なしなら従来位置 (短い鎖は完全に不変)。混んだ関数だけ伸びる
- 自己衝突除外は壊れた IR への防御ガード扱い (通常 dead — レビュー W2 でコメント修正)

### M2 へ引き継ぎ

- **折り曲げ**: 鎖の方向を変える (S/L 字)。M1 は直線維持
- **クロージャ陣の場所探し**: place_glyph_anchored_rings に同じ仕組みを横展開
- **警告フラグ**: オーバーラン (上限到達でも衝突残存) を IR に乗せて UI で可視化
- **精度テスト**: M1 の単体テストは「鎖 glyph 間距離が CHAIN_STEP 以上」止まり。
  M2 で「障害物との距離が CHAIN_AVOID_MARGIN 以上」の精密 assert に強化 (レビュー S2)
- 占有マップを構造体に切り出すかは、4 経路 (鎖 / 補助陣 / 通常 fan / 折り曲げ判定)
  から参照する段階 (M2) で再判断
