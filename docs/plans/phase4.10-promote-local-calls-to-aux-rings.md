# Phase 4.10 — ローカル呼び出しの補助陣昇格 (名前が見える)

## 出典

- オーナー要望 (2026-06-13、4.0.6 後半判定時): 「組み込みじゃない関数・メソッド呼び出しを
  補助陣に昇格して名前ラベル見えたらいい感じだと思う」

## 目的

外部呼び出し (`SummonGlyph`) は現状すべて同じ「小さな黒丸」として描かれる。
そのため `state_json` / `reload` / `helper` のような**同ファイル/同ワークスペースの
関数呼び出し**も、`println!` / `Vec::new` のような**組み込み呼び出し**と区別がつかない。
本計画では「**解決できるローカル呼び出し**」を補助陣 (AuxRing) に昇格し、4.0.6 後半で
入った補助陣ラベル機構をそのまま使って**名前が見える**ようにする。

## 判別の方針 (「組み込みじゃない」の定義)

優先順:
1. **ワークスペース呼び出しグラフで解決できるもの** (Phase 4.5 M2 前段の
   `workspace_index` の cross_edges / FileIndex の local_edges に出てくる相手 = 昇格対象)
2. **同ファイルで `function_index` に存在する名前** (現状 `resolve_local_target` が解決
   できるもの = 昇格対象。Phase 4.2 で構築済み)
3. **それ以外** (組み込み、外部クレート、解決失敗の `.method`) は従来どおり SummonGlyph

ローカル解決は **glyph 生成時には情報が無い** (function_index は serve 側で構築) 点が
重要 — IR 層に「ローカル呼び出しか」を判定する材料が無いので、設計をどこに置くかが
本計画の核。

## 設計案 (着手時に精緻化)

### A 案: serve 層で IR を後加工 (推奨候補)

- `magia-rust::parse_function` までは既存どおり SummonGlyph として作る
- `serve.rs::render_spell` で、関数索引 (good.functions) と call_target を照合し、
  解決できる SummonGlyph を **AuxRingKind::LocalCall** の AuxRing に置き換える
- 補助陣ラベルは関数名 (`Caster::cast` の `cast` 部分)
- 利点: parse_function / dataflow / metrics などの既存経路を無傷で残せる
  (ローカル解決は serve 層の知識 = 関数索引が必須なので筋が通る)
- 欠点: ir 層に「召喚と AuxRing が同位置・同役割」の重複が発生 — Sigil 種別の境界が
  ぼやける

### B 案: magia-rust 側で AuxRingKind::LocalCall を判定

- parse_function に「同ファイル内の関数名一覧」を渡し、ローカル解決できる呼び出しは
  最初から AuxRing に
- 利点: serve に後加工ロジックを持ち込まない、構造が一貫
- 欠点: ワークスペース横断解決 (4.5 M2 前段) を活かすには workspace_index を入力に
  繋ぐ追加配線が要る

### 共通の意匠 (案 A/B 共通)

- 補助陣の半径は SummonGlyph と同じ (14px) に縮小 — 通常の AuxRing (48px) より小さく、
  外部呼び出しからの「昇格」を見た目で示す。**新定数 `LOCAL_CALL_RING_RADIUS`**
- リング上の操作ドットは持たない (中身は不明 — 外部関数なので)。リング本体だけ
- 効果カテゴリ色は SummonGlyph と同じヒューリスティック (call_target からの推定)
- 補助陣ラベルは Phase 4.0.6 後半の `role_label` を拡張して関数名を出す

### M1 / M2 分割の案

- **M1**: 案 A で同ファイル解決 (function_index) のみ昇格 + ラベル。ワークスペース解決は
  M2 に送る。これだけで「`state_json` の中の `reload` が読める」効果が出る
- **M2**: ワークスペース呼び出し (cross_edges) も昇格。ファイル名サブラベル
  (`reload @ serve.rs`) も検討

## やらないこと (M1 スコープ)

- メソッドチェーンの中の `.method` の昇格 (レシーバ型不明で過剰昇格になる — Phase 4.8 の
  チェーン構造を壊さない)
- 関数本体の埋め込み展開 (定義ジャンプとは別軸 — クリックで `?pin=` 遷移は既存経路)
- 補助陣のラベルでなく **関数本体の縮小プレビュー** (M3+)

## 想定リスク

- **昇格しすぎ問題**: トップレベル関数が多いファイルだと「ほぼ全部が補助陣」になり、
  かえって読みにくい。同名関数が複数 impl にあるケースの解決ぶれ
- **chain glyph の特別扱い**: 鎖 (Chain edge) の途中の glyph は昇格させない方が良いか、
  全部昇格させるか — 判定で見てから決める
- **意匠の連続性**: クロージャ補助陣 (RING_RADIUS=48) と LocalCall 補助陣 (RING_RADIUS=14)
  の大小差が大きすぎないか
- **diff の同名解決**: NodeKey (Phase 3.1) が昇格前後で変わるため、既存の diff 比較で
  ノイズが出ないか確認 (4.8 でも同じ宿題あり)

## 判定ポイント (オーナー目視)

- 昇格後の見た目: 補助陣にラベルが乗って「読める」感じになっているか
- 通常の補助陣 (if/for/closure) との大小バランス
- chain の中の glyph を昇格させるかどうか
- ワークスペース解決 (M2) の必要性

## 受け入れ基準 (粗)

- [ ] 同ファイルの関数呼び出しが補助陣に昇格し、関数名ラベルが見える
- [ ] 組み込み呼び出し (`println!`, `Vec::new` 等) は従来 SummonGlyph のまま
- [ ] 既存テスト (metrics・dataflow・diff) を壊さない
- [ ] オーナー目視判定
