# Phase 4.8 — メソッドチェーンの鎖化 (召喚陣から召喚陣へ)

## 出典

- オーナー観察 (2026-06-12): 長いメソッドチェーンのホバープレビューを見て
  「メソッドチェーンは召喚陣から召喚陣に伸びたほうが良さそう」
- 文脈: serve.rs の `span_excerpt_collapses` 級の関数で、チェーンの各呼び出しが
  独立の召喚印としてリング外周にびっしり並び、連なりが図から読めない

## 目的

メソッドチェーン (`x.iter().skip(1).filter(…).map(…)`) を、リングに係留された
**先頭の召喚陣から後続の召喚陣へ鎖が伸びる**形で描く。
「値が手渡されながら変換されていく」というチェーンの意味論を図に出す。

## 現状の構造 (実装済みの土台)

- `collect_calls_in_stmt` (magia-rust/summon.rs) はチェーンをフラットに展開し、
  各呼び出しを独立の `CallSite` → `SummonGlyph` にする
- 配置は glyph 単位でリング外周に等角度散布 (1.8 の衝突回避込み)
- spec v0.2 §4.3 で EdgeLayerData は再設計済み — Edge 種別の追加は素直にできる

## 設計案 (着手時に精緻化)

1. **抽出 (magia-rust)**: `ExprMethodCall` のレシーバ入れ子から chain を検出し、
   `CallSite` に `chain_id` + `chain_index` を付与。`x.a().b().c()` →
   a(0) → b(1) → c(2)。関数呼び出しが引数中に挟まるケース
   (`x.a(f()).b()`) では f はチェーンに含めない (別の独立 glyph)
2. **IR**: SummonGlyph 間の `EdgeKind::Chain` を新設 (spec 追補)。
   chain の隣接ペアを結ぶ
3. **レイアウト (magia-core)**: チェーン先頭だけをリングに係留し、後続は
   外向き放射方向に数珠つなぎ配置。既存の衝突回避 (1.8) と統合。
   viewBox はチェーン長に応じて拡張
4. **描画 (Vue)**: チェーンエッジは細線 (制御フロー線と区別できる意匠)。
   ホバープレビュー・インスペクタは既存の glyph 単位のまま動く

## 判定ポイント (オーナー目視)

- 鎖の伸びる向き・glyph 間隔・エッジの意匠 (細実線か点線か)
- 長いチェーン (5+) での場所取りと viewBox の収まり
- 効果カテゴリ色 (各リンクの色は呼び出しごとに変わる) と鎖の視認性の両立

## 想定リスク

- 超長チェーン (10+) の放射配置が他の補助リング・チップと衝突する
  → 途中折り返し (螺旋) は判定を見てから
- `?` / `.await` 混じりのチェーン (`x.a()?.b().await`) の chain 境界判定
- ベルカ式 (フロー集計) への反映は本計画では対象外 (集計値は不変)

## 受け入れ基準 (粗)

- [ ] チェーンが先頭 glyph からの数珠つなぎで描かれ、エッジで連結される
- [ ] チェーン外の呼び出し (引数内の関数呼び出し等) は従来どおり独立
- [ ] 既存テスト + チェーン抽出の単体テスト + 描画の e2e
- [ ] オーナー目視判定

## 実装結果メモ — M1 (2026-06-12)

### 実装内容

- **抽出**: chain_members がレシーバ入れ子を Try/Await/Paren/Reference 透過で辿り、
  基底の ExprCall (`f(x).a()`) も鎖の先頭に含める。最外殻の visit が1回で処理し、
  **実行順 (最奥→外)** で CallSite 化 (glyph 生成順が旧実装の syn 走査順から変更 [break])。
  引数・基底式は手動再帰で独立 glyph のまま。`as` キャストは透過しない (M1 スコープ外)
- **IR**: EdgeKind::Chain 新設。鎖後続は先行 glyph から Chain、先頭だけリングから
  ControlFlow。不変条件は「glyph は係留 Edge (ControlFlow|Chain) 入次数ちょうど1」へ [break]
- **layout**: place_chain_glyphs — 後続を先頭から外向き (係留リング中心→先頭) に
  CHAIN_STEP=40 間隔の直線配置。**鎖どうし・他要素との衝突回避は M1 では未実施**
  (直線をまず判定に出す)。循環ガードあり
- **描画**: EdgeIr.kind (control_flow|chain)、EdgeLine が鎖を細線 0.6 で描き分け
- **diff**: 鎖後続も diff の木に入る (レビュー Critical 1 — 係留 Edge の追従。回帰テスト済み)

### 判定待ち / M2+ 候補

- 鎖の間隔 (CHAIN_STEP=40)・細線 0.6 の意匠 → オーナー判定
- 長い鎖の衝突回避 (他の fan glyph・チップとの重なり) — 判定を見てから
- 鎖の途中折り返し (10+ 連鎖)、NodeKey への鎖内位置の安定キー (diff の同名メソッド対応)
- `as` キャストの透過 (必要になったら Expr::Cast を chain_members に足す)
- 4.7 (魔力回路) との合流: 点Pが鎖を伝う表現の土台はこれで完成

## 実装結果メモ — M2: コールバック補助陣 (2026-06-12)

**M1 は判定合格 (「鎖チェーンめっちゃいいー！」)。M2 はオーナー要望
「コールバックを補助陣にするともっとよさそう」を同日実装。**

### 実装内容

- AuxRingKind::Closure 新設 — 召喚の引数のクロージャが glyph に係留される補助陣になる
  (glyph が子リングを持つ最初の形)。label = `|params|`、リングホバーでクロージャ全体の
  原文が見える (guard_location 流用)
- CallSite<'ast> 化 + 全経路 default 再帰廃止 (クロージャ本体は補助陣側で展開 —
  本体内の制御構造・チェーン・さらにそのクロージャも再帰的に成立)
- layout: place_glyph_anchored_rings — 波 (wave) 配置。波ごとに入れ子リング展開 +
  wave 限定 fan + 鎖。兄弟クロージャ (rayon::join 形) は扇形分散 (レビュー W1)
- transcript の補助リング内訳に「クロージャn」追加 (insta 更新)

### 判定待ち / M3+ 候補

- クロージャ陣の距離 (RING_GAP) と兄弟分散角 (0.9 rad) の意匠
- 鎖 + クロージャ陣が混む関数での衝突回避 (M1 から継続の宿題)
- 即時呼び出しクロージャ `(|x| f(x))(val)` は平坦化のまま (既知の非対称、稀)
- 凡例にクロージャ陣の説明追加 (意匠が落ち着いたら)
