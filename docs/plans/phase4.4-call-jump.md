# Phase 4.4 — 呼び出しジャンプ (ピン遷移を関数間の地図移動に)

## 出典

- オーナー方針 (2026-06-11): RTS の「主戦線移動」感を「呼び出し記号をクリック → 焦点が呼び出し先に移動」に翻訳
- 親計画: Phase 4.1 (ピン中心ビュー) + Phase 4.2 (近接度モデル) + Phase 3.4 (データフロー IR)

## 目的

フォーカス関数の魔法陣内に描かれた「呼び出し記号」をクリックすると、その呼び出し先関数にピンが移動するインタラクションを追加する。
4.1 で実装した「周辺盾クリックでピン遷移」と合わせ、**「魔法陣の内側からの遷移」「外側からの遷移」両方が成立**する状態にする。

ユーザ体験上は「コードを読む」が「地図を歩く」に変わる。

## スコープ

### やること

- **呼び出し記号の特定**: 4.1 でレンダされたフォーカス魔法陣内で、Phase 3.4 のデータフロー IR から得た「同ファイル内の関数呼び出し」に対応する SVG 要素に `data-call-target="<fn>"` 属性を埋め込む
- **クリックハンドラ**: 該当要素クリックで `?pin=<target>` に遷移 (4.1 のピン遷移と同じ経路)
- **ホバー強調**: 呼び出し記号にホバーすると周辺盾の対応関数もハイライト (両者の対応関係を視覚化)
- **逆方向 (callee → caller)**: 周辺盾に「← 呼ばれる」マークを付け、これをクリックすると caller にピン遷移
- **アニメーション**: 4.1 のピン遷移アニメをそのまま流用。`prefers-reduced-motion` 尊重
- **静止画 (4.3)**: `--with-call-arrows` フラグで「焦点 → 周辺の呼び出し関係」を矢印で overlay

### やらないこと

- ファイル外への呼び出しジャンプ (4.5 ワークスペース俯瞰)
- trait method の動的ディスパッチ解決 (best effort)
- 呼び出しグラフの全体俯瞰ビュー (これは 4.5 の延長で別途検討)

## 設計上の判断

### `data-call-target` を SVG に埋め込む

クライアント JS で IR を持ち回るより、レンダ時に SVG 属性として埋めておく方が単純。
- サーバ側で IR 知識を完結させられる
- クライアントは「`data-call-target` 付き要素にクリックリスナー」だけ

### Phase 4.2 の近接度との関係

呼び出し関係は 4.2 で既に近接度に算入されている。本計画は「近接度の根拠を UI 上で可視化する」役割。
新規モデルは作らず、4.2 の `ProximityIndex` から呼び出しエッジだけ取り出す。

## 受け入れ基準

- [ ] フォーカス魔法陣内の呼び出し記号クリックでピンが呼び出し先に移動
- [ ] 呼び出し記号ホバーで対応する周辺盾がハイライト
- [ ] 静止画 `--with-call-arrows` で呼び出し矢印が overlay される
- [ ] 既存テスト + 呼び出しクリックの統合テスト

## 実装ステップ (粗粒度)

1. `magia-core` のレンダ層で呼び出し記号に `data-call-target` 属性を付与
2. フロント JS で属性付き要素のクリック → 4.1 ピン遷移ハンドラ
3. ホバー強調 (CSS + JS でクラス swap)
4. 周辺盾に caller マーク
5. 静止画 `--with-call-arrows`
6. テスト + Stage 1/2 + 完了処理

## 想定リスク

- **呼び出し記号の SVG 要素特定**: 既存レンダパイプラインで「どの SVG ノードがどの呼び出しに対応するか」のマッピングが必要。Phase 3.4 のレンダ層が data-* 属性をどう扱っているか実装時に確認
- **trait method**: 静的に解決できない呼び出しはクリックしても遷移先がない。記号は描いて hover で「動的ディスパッチ」とだけ表示する fallback
