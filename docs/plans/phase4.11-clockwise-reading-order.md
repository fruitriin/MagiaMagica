# Phase 4.11 — 陣の描画方向を時計回りに (読みやすさの整合)

## 出典

- オーナー要望 (2026-06-13、4.0.6 後半・入口サイン判定時):
  > 別件なんだけど陣の描画方法を反時計回りから時計回りにしたほうが読みやすいかもしれない。
  > なぜなら横書きは左から右へ、縦書きは上から下へ読みので

## 目的

メインリングの操作ドット配置・補助陣のループ記号・シグネチャ円弧・入口サイン・
ベルカ三極などの **進行方向を時計回り** に統一する。横書き (左→右) と縦書き (上→下)
の自然な視線移動に合わせ、初見の読みやすさを優先する。

## 起点の選択 (着手時に確定)

候補 A: **12時起点・時計回り**
- 12時 → 3時 → 6時 → 9時 → 12時
- 時計を読む経験と一致、起点が「上 (始まり)」で自然
- 横書き読みの「左」から始まらない違和感はあるが、SVG 中心配置では上が自然

候補 B: **9時起点・時計回り**
- 9時 → 12時 → 3時 → 6時 → 9時
- 横書きの「左」起点と一致 (オーナーの理由付けに最も近い)
- ただし時計の起点 (12) と違い、視覚的に「9時から始まる」が直感的でないかも

**推奨**: **A (12時起点・時計回り)**。理由 — (1) シグネチャ円弧が現状 9時 → 上 → 3時の上半円なので、上半円の中央 (12時) が起点だとシグネチャと進行方向の連続性が取れる、(2) 時計の起点規約 (12時から開始) と一致。

決定は M1 着手前にオーナー判定で確定する。

## 影響範囲 (洗い出し)

### 1. レイアウト (magia-core)

- `midchilda.rs::screen_position` / 操作ドットの角度割付 — `3時起点・反時計回り` の規約を改める
- `layout/mod.rs::place_main_ring` — `outward = 3時方向` で aux ring 配置の基準が決まる。
  起点が変わるなら見直す
- spec §6.1.2 (操作ドット配置) — 起点と回転方向の規約条文の改訂

### 2. 描画 (Vue)

- `SymbolMark.vue::loopPoints` — ループ三角は反時計回りで「左向き」。時計回りでは
  「右向き三角」に
- `EntrySign.vue` — 矢じり方向を反転 (時計回り = 12時 → 3時方向)、三角 ▷ の向きも検討
- `SignatureArc.vue` — textPath の `arc_path` (現状 9時 → 12時 → 3時) はそのままでも
  読みやすいが、時計回り起点との関係性は再考
- `irToSchema::buildRingLabel` — 補助陣ラベルの弧 (現状上半円) も同上
- `BelkaCircle.vue` — 三極の配置 (生成・変換・消費) は時計回り順に並んでいるか確認

### 3. 凡例

- `SymbolLegend.vue` の説明文 — 「3時起点・反時計回り」を「12時起点・時計回り」に書き換え、
  入口サインの説明も同期
- ループ三角の説明 — 「反時計回りに周回」を「時計回りに周回」に

### 4. ベルカ式

- `belka.rs` の三極配置 (Vue 側 BelkaCircle.vue の POLE_STYLE) — 時計回りの読み順に
  並んでいるか?

### 5. 既存判定済みの意匠

- 1.6 (中規模意匠)・1.8 (衝突回避意匠) は「反時計回り」前提の意匠判定だった。**時計回り化は
  M0 で判定を再取得する必要あり** — 新規の判定 (オーナー目視) として組み直す

### 6. テスト・ゴールデン

- スナップショット (insta) は全て差し替わる
- e2e のループ三角向き検証など、向き依存のアサートは更新

## 設計案 (M1/M2 分割)

**M1 (中規模・破壊的変更)**:
- 操作ドット配置を時計回りに (起点はオーナー確定後)
- ループ三角を右向きに
- 入口サインの矢じりを反転 (12時 → 3時方向)
- 凡例の説明文を更新
- 既存テスト・スナップショット差し替え

**M2 (判定後の意匠ブラッシュ)**:
- シグネチャ円弧・補助陣ラベル弧の向きを検討
- ベルカ三極の並び順 (現状で良ければ無修正)
- 入口サイン ▷ の向き・撤去判断

## やらないこと

- 「両方向対応」(`?direction=clockwise` のような設定可能化) — 意匠の根幹は1つに統一する
- spec のレジストリ規約 (時計回り = magic circle の文化的伝統 vs 反時計回り = 数学的慣例) の
  議論。本計画は「読みやすさ」を理由に時計回りに倒すと決め打つ

## 想定リスク

- **判定済み意匠の再判定が必要**: 1.6 / 1.8 で合格した「反時計回り前提の意匠」が
  時計回りで違和感を出す可能性。M1 の素材送付の必須項目
- **影響範囲が広い**: 上記 6 領域に手が入る大改修。1 サイクルでは収まらない
- **spec の改版**: 仕様が「反時計回り」と書かれているため spec-v0.3 → v0.4 への
  改版で吸収する形が筋
