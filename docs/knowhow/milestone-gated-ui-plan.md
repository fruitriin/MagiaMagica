# 目視判定ゲート付き UI 計画の運用

> Phase 4.0.5 (Vue 基盤、M0〜M6 + 判定ゲート5回) で確立したプロセス知見。
> コードパターンは viteplus-bun-frontend-bootstrap.md / minimal-dev-server-pattern.md 側。

## 発見した知見

### 判定ゲートは「サイクル分割の強制」ではなく「進行の同期点」

- 計画では「1サイクル = 判定待ちゲートまで」としたが、オーナーがアクティブに会話している間は
  **同セッション内で 判定 → 次マイルストーン を連続で回せた** (M1〜M6 を1日で完走)。
  ゲートの本質は「NG 残置で次へ進まない」ことであり、待ち時間ではない
- 判定で**追加要望**が出たら: 今のマイルストーンの修正指示 (即対応) と「別スコープで OK」の
  要望 (即計画ファイル化 + TODO 登録) に仕分けてサイクルを止めない。後者は出典
  (「オーナー要望 YYYY-MM-DD、Mx 判定時」) を計画書に残すと後で文脈が辿れる
- 判定の口頭決定 (例: バイナリサイズは気にしない) は**その場で CLAUDE.repo.md / 計画書の
  受け入れ基準に反映**する。Feedback.md の判定待ち項目も同時に更新

### URL に全状態を乗せると判定素材が安価になる

- UI 状態 (関数選択・レイヤー可視性・透明度・様式) を URL クエリに双方向同期させた結果、
  **「状態を再現したスクショ」がヘッドレス Chrome の URL 直開き1コマンドで撮れる**
  (`chrome --headless --screenshot=out.png 'http://…/?fn=x&layers=a,b&op=a:0.5'`)。
  判定素材・ドキュメント・バグ再現の全部で効く。SPA の状態設計を決めるとき
  「この状態は URL で再現できるか」を判断軸に入れる価値がある

### 基盤入れ替え (機能等価な移植) は潜在バグの炙り出し装置

- 素 JS → Vue の「振る舞いを変えない移植」は、移植自体が**既存機能の全数再検証**になる。
  SSE が Phase 2.1 から配信されていなかった潜在バグは、M2 の移植 E2E で初めて発覚した
  (旧 UI では「初回ロードだけで画面が出る」ため誰も気づかなかった)
- 移植中に「動かない」を見つけたら、**新基盤側のバグより先に旧実装の潜在バグを疑う**。
  切り分けは「旧経路 (curl 直叩き等) で同じ操作を再現する」が最短
- 移植で見つけた旧バグは移植計画の中で直してよい (関心が同じ機能のブロッカーのため) が、
  「なぜ既存テストで捕まらなかったか」を Feedback の問題記録に残し、テスト定型を knowhow 化する

### 画素等価検証の方法論 (Phase 4.0.7 で確立)

「同じ見た目のまま実装を入れ替える」移行の検証手順:

1. **git stash で新旧を切り替え、同一ツール・同一条件でスクショを撮る** (untracked の新規ファイルは stash されないが、参照元が旧版なら使われないので問題ない)
2. **同版2回撮りのベースラインノイズを先に測る** — 0 なら撮影は決定論的で、差は全て実差
3. `magick compare -metric AE` の絶対値だけで判断しない: **fuzz 5% でほぼ消える差は AA (アンチエイリアス) ゆらぎ**。
   `-roll +1+0` のシフトテストで差が激増するなら位置ズレではない (局所的な色差)
4. 残差は connected-components で位置を特定し、最大クラスタを拡大して目視 — 文字サイズ以下なら視覚的等価
5. 既知の落とし穴: **要素の描画順 (z-order) を変えると重なり箇所のピクセルが変わる** (スキーマに出現順 `z` を持たせて再現する)。
   `opacity: 1` でも style で明示すると stacking context が生まれ AA が変わりうる (既定値はスタイル無しに)
6. Vue 生成 DOM と innerHTML パースの間には fuzz 5% 未満の AA 差が原理的に残る — 「視覚的等価」を受け入れ基準にする (計画書にも明記しておく)
7. **撮影条件は起動引数まで完全に揃える** (Phase 4.0.9 で再確認): サーバ起動の
   相対/絶対パスの違いがヘッダのファイルパス表示の折返しを変え、**ページ全体が
   縦シフトして全面 diff になる**。比較ペアは同じ cwd・同じ引数で起動したサーバから撮る
8. レイアウト/描画ロジックを段階的にリメイクするときの境界判断 (Phase 4.0.9):
   「どこに何があるか (座標・配置・衝突回避済み円弧)」は元実装側に残し、
   「どう描くか (記号の頂点・線の端点)」を新実装に移すと、二重実装なしで
   両者が同じ絵を出せる — 境界スキーマはこの分担を型で表現する
9. **新旧実装が並存する過渡期間はクロス検証テストを書く** (Phase 4.0.9): 同一入力を
   両経路で出力し、座標を正規化 (丸め桁を揃える + 負のゼロ対策) して集合比較する。
   「規則のコピーがいつの間にかずれる」事故を旧実装の削除まで防げる。
   追加直後に -0.0 の JSON 混入と、レビューでは言語間の手書き定数コピー
   (TS 30 vs Rust 26.0) も検出された — **言語をまたぐ定数・規則の複製には
   機械検証を必ず添える**

### Playwright の toBeHidden は「存在しない」でも通る

- 実装入れ替えで DOM 構造が変わると、旧セレクタの `toBeHidden()` が**要素消滅により偽陽性で通る**。
  「先に toBeVisible で存在を確かめてから toggle → toBeHidden」の2段にして偽陽性を防ぐ

### サイクル中の追加要望は「実装→素材→判定」の小ループで受ける (Phase 4.1)

- オーナー在席中の試遊からは**連鎖的に要望が出る** (4.1 ではインスペクタ → 呼び出し式 →
  ホバー2層化 → 補助リング条件、の4連鎖)。1要望 = 1小ループ (実装 + ゲート + コミット +
  素材送付) で受けると、各要望が前の判定結果の上に積めて手戻りがない
- 大きい要望 (意匠の作り直し級) は**その場で実装せず計画ファイル登録だけ**で受ける
  (4.3.5 結節点シンボル)。「いつやるか」をオーナーに指定してもらい TODO の依存関係に固定する
- Progress.md には追加要望を A/B/C/D... のセクションで積む — 本体チェックリストを
  汚さず、完了処理時に計画 memo へまとめて転記できる

### インタラクティブ SVG では「装飾とインタラクション対象」を pointer-events で分離する (Phase 4.1)

- 図に重なる装飾 (シグネチャ円弧テキスト・制御記号) は**クリック/ホバー判定を奪う**。
  装飾コンポーネントには `pointer-events: none` を必ず付ける — 「これは読むだけの要素か、
  触る要素か」をコンポーネント新設時に1回判断する
- 線だけの要素 (fill: none の円) をホバー対象にするなら `fill: transparent` で面全体を
  判定にする。後に描画される要素 (上のレイヤー) のホバーが自然に優先される
- マウス直近に出すプレビューは自身に `pointer-events: none` — 出た瞬間に mouseleave が
  発火してチラつくのを防ぐ

## プロジェクトへの適用

- Phase 4.0.7 / 4.0.9 (SVG パーサ → IR ビルダ) も「振る舞いを変えない入れ替え」— 同じ運用
  (golden 先取り → 移植 → 等価確認 → 判定) を適用する
- 4.1 のピン中心ビューも `?pin=` URL 同期前提なので、判定素材は URL 直開きで作る

## 注意点・制約

- 連続判定はオーナーの在席に依存する。不在時は計画どおり「判定待ちでサイクル終了」に
  戻す (Progress に「判定待ち」を明記してコミットまで済ませる)
- 「画素単位等価」の厳密な検証は目視 + golden テキスト比較で代替した。ピクセル diff が
  本当に必要になったら Playwright の toHaveScreenshot を検討 (フォントレンダリングの
  環境差に注意)

## URL を唯一の状態源にする設計の穴 (細部修正 2026-06-12)

- **「URL の値を常時主張する」watch は複数タブで綱引きになる**: ?file= のような
  サーバ側状態を URL から復元する watch を「不一致なら常に POST」で書くと、
  異なる ?file= を持つ2タブが SSE 越しに無限 POST 合戦をする。**復元は
  1ページロードにつき最初の不一致の1回だけ**にし、以後の不一致は「外部が
  切り替えた」とみなしてサーバに追従 (requestedFile をサーバ値で上書き) する
- 行単位 syntect (HighlightLines + styled_line_to_highlighted_html) は
  パースに改行込みで渡し (構文状態の引き継ぎ)、HTML からは改行を落とす
  (1行1要素)。行リストの :key は内容でなくインデックス (`}` だけの行など
  同一内容行が普通に複数あるため)
- ソースの「全体表示 + フォーカス範囲強調」は /state (reload 時1回構築) に
  行配列で同梱し、関数切替はクライアントの範囲差し替えだけにする —
  切替のたびにソースを取り直さない
