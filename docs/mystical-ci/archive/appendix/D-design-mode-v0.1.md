# Mystical CI Appendix D: 設計支援モードと式駆動設計 v0.1

> **本文書の位置づけ**
> 本文書は『Mystical CI 仕様書 v0.1』および各 Appendix に続く補遺 D である。これまで「コード → 魔法陣」の順方向で議論してきたツールを、「求める性質 → 魔法陣 → コード設計」の逆方向にも使う**双方向ツール**として再定義する議論を集約する。
>
> 本文書は本ツールの**位置づけを根本的に拡張**するため、Phase 2 以降ノートに対する重要な追補となる。Phase 5 以降の実装スコープに影響する。

---

## 1. ツールの双方向性

### 1.1 二つのモード

本ツールは、これまで「コードを解析して魔法陣を生成する」順方向の使い方で議論されてきた。これを **Analyzer Mode** (解析者モード) と呼ぶ。

これに対し、本文書では新たに **Designer Mode** (設計者モード) を提案する。求める性質や課題設定から、それに該当する魔法陣を選び、その骨格に基づいてコードを設計する逆方向の使い方である。

```
[Analyzer Mode]  Code → Analysis → Magic Circle
[Designer Mode]  Desired Property → Magic Circle → Code Skeleton
```

両モードは同じ視覚言語 (式の体系) を共有し、双方向に行き来できる。Analyzer で診断した結果から、Designer で改善案を構築する、というワークフローが自然に成立する。

### 1.2 モードの切替

CLI レベルでの想定インターフェース:

```bash
# Analyzer Mode (Phase 1 以来の使い方)
$ mystical render src/state/reducer.rs::update

# Designer Mode (新規)
$ mystical design --property fault-tolerance --domain api-handler
$ mystical design --style alchemy --output-skeleton src/skeleton/
$ mystical design --recipe "state machine with refinement types"
```

Designer Mode は対話的な体験が向く。「どんな性質が欲しい?」「適用するドメインは?」「既存の式から選ぶか、組み合わせるか?」といった問いに答えることで、最終的な式と骨格コードが決定される。

### 1.3 価値の乗算

双方向ツールになることで、ツールの価値は二倍ではなく**乗数的に**増える。Analyzer の出力 (現状の式) と Designer の入力 (目標の式) を比較できることで:

- 現状の式が目標と一致 → 設計が意図通り
- 現状の式が目標と異なる → 設計とのズレを可視化
- ズレを埋めるリファクタリング提案 → 式の変換手順

つまり、本ツールは「**式の変換器 (Style Transformer)**」としても機能できる。これは Phase 5+ の野心的な機能だが、双方向性さえ確保すれば理論的には到達可能である。

## 2. 異質式適用 (Cross-Paradigm Application)

### 2.1 概念

各ドメインには「当たり前」とされる式が存在する。状態管理ならベルカ式、API ハンドラならうみねこ式、リソース管理なら黒執事式、といった慣習である。

**異質式適用**とは、この当たり前を意図的に破り、別の式を当てることで新しいパラダイムと優位性を引き寄せる設計手法である。

これは単なる奇抜さの追求ではなく、既存の技術選定の**暗黙の前提**を疑い、別の選択肢が持つ優位性を意識的に取り込む実践である。技術史上、多くのブレイクスルーは「当たり前を疑った異質適用」から生まれている。

### 2.2 具体例

以下、典型的なドメインに対する異質式適用の例を整理する。

#### 例 1: 状態管理に Alchemy式

**当たり前の式**: ベルカ式 (3-fold Convergent) — Reducer パターン
**異質適用**: Alchemy式 (5-fold Balanced) — 契約と等価交換

ベルカ式は `State + Action → NextState` の三項変換だが、Alchemy式を当てると各遷移に前後条件・不変条件が課される。状態遷移は「正しい遷移として証明可能なもののみ許容される」契約構造になる。

**引き寄せられる優位性**:
- 型レベルで不正な状態遷移を排除
- Refinement Types ベースの安全性
- 状態爆発の制御 (有効状態空間が型で絞られる)

**参照実装**:
- Elm (型安全な状態管理)
- F* (FStar) の状態モデル
- TLA+ で仕様化された状態機械

**得られないもの**:
- Reducer の単純さ
- 動的な状態構造への柔軟性

#### 例 2: API ハンドラに RWBY式

**当たり前の式**: うみねこ式 (4-fold Orthogonal) — 4xx/5xx の離散分類
**異質適用**: RWBY式 (6-fold Divergent fractal) — フラクタル並列

うみねこ式はステータスコードや分類による排他的ディスパッチだが、RWBY式を当てると各エンドポイントが独立した並列ワーカーになり、ハンドラ間の独立性と並列性が主軸になる。

**引き寄せられる優位性**:
- Actor モデル的な分散ハンドラ
- 1エンドポイントの障害が他に伝播しない
- 水平スケーリングの自然さ
- 自己相似的な耐障害性 (Supervisor Tree)

**参照実装**:
- Erlang/OTP, Elixir Phoenix
- Akka HTTP

**得られないもの**:
- ステータスコード階層の単純な扱い
- 中央集権的な認証/認可の容易さ

#### 例 3: リソース管理に Vesperia式

**当たり前の式**: 黒執事式 (5-fold Divergent) — 不可逆契約 (Drop)
**異質適用**: Vesperia式 (4-fold Organic) — 有機伝播

黒執事式の所有権/Drop はリソースの確実な解放を契約として強制するが、Vesperia式を当てるとリソース管理が横断的関心事として実装され、リアクティブな取得/解放サイクルになる。

**引き寄せられる優位性**:
- GC 言語の柔軟なリソース管理
- Weak references による循環参照の自然な解決
- リソース使用の動的な観測と調整

**参照実装**:
- Java/C# の AutoCloseable + WeakReference
- Reactive Programming のリソース管理 (RxJava)
- Python の `with` ステートメント + ファイナライザ

**得られないもの**:
- コンパイル時のリソース漏洩検出
- ゼロコスト抽象化

#### 例 4: DI コンテナにベルカ式

**当たり前の式**: Fate式 (8-fold Convergent + core) — ポート/アダプタの全方位
**異質適用**: ベルカ式 (3-fold Convergent) — 三項変換

Fate式の DI は多ポートで柔軟だが、結合点が多くなりがち。ベルカ式を当てると依存性が極端に絞られた、最小限の三層 DI 構造になる。

**引き寄せられる優位性**:
- Three-tier architecture の純粋形
- テスタビリティの極大化
- 設定ファイル不要、手動 DI の透明性

**参照実装**:
- 手書きの三層構成 (Presentation/Domain/Infrastructure)
- Functional Core, Imperative Shell (Gary Bernhardt)

**得られないもの**:
- 大規模アプリケーションの柔軟性
- アスペクト指向との統合

#### 例 5: 並列処理に Alchemy式

**当たり前の式**: RWBY式 (6-fold Divergent fractal) — フラクタル並列
**異質適用**: Alchemy式 (5-fold Balanced) — 契約と等価交換

並列処理に契約を持ち込むと、並列性が「等価交換可能な変換」として表現される。Software Transactional Memory (STM) の発想である。

**引き寄せられる優位性**:
- 並列計算の合成可能性
- ロックフリーな共有状態
- 数学的に証明可能な並行性

**参照実装**:
- Haskell STM
- Clojure の Software Transactional Memory
- Concurrent ML

**得られないもの**:
- ベアメタル性能
- 細粒度のスレッド制御

#### 例 6: ビジネスロジックに夜天の書式

**当たり前の式**: ミッドチルダ式 (4/8-fold Balanced) — 構造化階層
**異質適用**: 夜天の書式 (Meta-recursive) — 自己参照メタ

ビジネスロジックにメタプログラミングを持ち込むと、業務ルール自体がデータとして表現され、ルールエンジンや DSL として実装される。

**引き寄せられる優位性**:
- ルール変更がコード変更ではなくデータ変更になる
- 非エンジニアによるルール記述の可能性
- A/B テストや動的な業務フロー変更

**参照実装**:
- Drools (Java)
- Excel ベースの業務ルール
- LISP の業務 DSL

**得られないもの**:
- ロジックの静的な解析容易性
- パフォーマンス予測の容易さ

### 2.3 異質式適用の方法論

異質式を採用する際の手順:

**Step 1: 当たり前の式の特定**
そのドメインで慣習的に使われている式を特定する。これは Analyzer Mode で既存実装を解析することで把握できる。

**Step 2: 当たり前の限界の言語化**
慣習の式が持つ既知の制約・限界をリストアップする。「この限界が問題になっている」と感じる箇所が、異質適用の動機となる。

**Step 3: 求める優位性の言語化**
解決したい問題から、求める優位性を明示する。fault-tolerance、型安全性、合成可能性、柔軟性など。

**Step 4: 候補となる式の選定**
Appendix A のカタログから、求める優位性に対応する式を選ぶ。

**Step 5: パイロット実装**
小さなモジュールで異質式を試す。全面採用の前に、その式が実際にドメインに馴染むかを検証する。

**Step 6: 評価**
パイロット実装で得られた優位性と、失われたものをリストアップ。トレードオフを評価する。

**Step 7: 採用判断**
全面採用、ハイブリッド (式の合成)、または棄却の判断を行う。

## 3. 式駆動設計 (Style-Driven Design)

### 3.1 概念

異質式適用を一般化すると、**設計を式の選択から開始する**手法に到達する。これを式駆動設計 (Style-Driven Design, SDD) と呼ぶ。

```
Traditional Design Flow:
  Requirements → Architecture → Modules → Functions → Code

Style-Driven Design Flow:
  Requirements → Style Selection → Magic Circle Sketch → Code Skeleton → Code
```

式駆動設計では、最初に求める性質に対応する式を選び、その式の骨格 (魔法陣のスケッチ) を描き、骨格に従ってコードを実装する。

### 3.2 SDD の利点

**設計の意図が視覚的に明示される**: ベルカ式を選んだなら「これは三項変換である」が図像として明示される。後から見ても設計意図が伝わる。

**パラダイムの一貫性が保たれる**: 式に従って設計するため、ドメイン内の一貫性が自然に維持される。

**異質適用が制度化される**: 式を意識的に選ぶプロセスがあるため、「当たり前」を疑う機会が組み込まれる。

**チームの設計議論が構造化される**: 「この機能はベルカ式か、Alchemy式か?」という問いが、設計議論の出発点になる。

### 3.3 SDD の限界

**式が固定される弊害**: 一度選んだ式に縛られすぎると、ドメインの実情に合わない実装になる可能性がある。

**創造性の限定**: 既存の式カタログから選ぶことになるため、本当に新しいパラダイムは生まれにくい。

**コミュニケーションコスト**: チームメンバー全員が式の体系を理解する必要がある。

これらの限界は、ツールが「式を強制する」のではなく「式を提案する」立場に留まることで緩和される。Phase 1 仕様書の設計原則「ツールは判決を下さず、注意を誘導する」は、SDD にも適用される。

## 4. 式のレシピ集

### 4.1 レシピの形式

式駆動設計を実用化するには、性質と式のマッピングを**レシピ集**として整備する必要がある。

各レシピは以下の構造を持つ:

```yaml
recipe_id: fault-tolerance-via-fractal
desired_property: 
  - fault-tolerance を最大化
  - 個別障害の局所化
  - 自己相似的なスケーラビリティ
recommended_style: RWBY
applicable_domains:
  - API handlers
  - background workers
  - data processing pipelines
expected_benefits:
  - 1ノード障害が他に伝播しない
  - 水平スケーリングが自然
  - Supervisor pattern による自己治癒
trade_offs:
  - 中央集権的な制御が困難
  - 状態共有のコスト
reference_implementations:
  - Erlang OTP
  - Akka Actors
  - Phoenix LiveView
code_skeleton: "src/skeletons/rwby-actor-tree.rs"
```

### 4.2 主要レシピの初期集合

ツールに同梱される初期レシピの候補:

**型安全性最大化レシピ** → Alchemy式 (Elm, Haskell)
**fault-tolerance レシピ** → RWBY式 (Erlang OTP)
**横断的関心事分離レシピ** → Vesperia式 (Spring AOP)
**契約による設計レシピ** → Alchemy式 (Eiffel, F*)
**マイクロカーネルレシピ** → Fate式 (NestJS, Spring)
**リアクティブストリームレシピ** → Vesperia式 + RWBY式 ハイブリッド (RxJS)
**メタプログラミング DSL レシピ** → 夜天の書式 (Racket, Rebol)
**Functional Core レシピ** → ベルカ式 + Alchemy式 (Elm, Bernhardt スタイル)
**Event Sourcing レシピ** → ベルカ式 (Command/Event/State の三項)
**Saga パターンレシピ** → ベルカ式 + 黒執事式 ハイブリッド
**Pipes and Filters レシピ** → 直列結合トポロジー
**Pub/Sub レシピ** → Vesperia式 + ハブ&スポーク

これらは Christopher Alexander のパターン言語、GoF のデザインパターン、Fowler の EAA パターンに連なる**視覚的パターン言語**となる。

### 4.3 コミュニティによる拡張

レシピ集は標準ライブラリではなく、コミュニティで継続的に育てる仕組みを想定する。

```
mystical-recipe-microservices  // マイクロサービス系レシピ
mystical-recipe-game-engine    // ゲームエンジン系レシピ
mystical-recipe-ml-pipeline    // 機械学習パイプライン系レシピ
mystical-recipe-embedded       // 組み込み系レシピ
```

クレートとして配布され、`Cargo.toml` で依存指定する。

## 5. 式の合成 (Style Composition)

### 5.1 概念

実際のコードベースは、単一の式で記述されることは稀である。多くの場合、複数の式が組み合わさってひとつのシステムを構成する。これを**式の合成 (Style Composition)** と呼ぶ。

合成のパターンは大きく以下に分類できる:

**並列合成**: 異なるモジュールが異なる式を持ち、結合点で繋がる (Appendix B で議論済み)

**入れ子合成**: 大きな式の構造の中に、別の式が部分的に埋め込まれる

**重ね合わせ合成**: 同じモジュールが複数の式の性質を同時に持つ (ハイブリッド)

**変換合成**: 時間の経過とともに、式が別の式へと変化する (動的合成)

### 5.2 ハイブリッド式の例

#### Hybrid 1: ベルカ + Alchemy = 契約付き Reducer

Reducer に前後条件と不変条件を加えた構造。Elm の Update 関数、F* のステートモナド、Refinement-typed Redux などがこれに該当する。

```
ベルカ式の骨格 (State + Action → NextState)
  + Alchemy式の契約 (各遷移の前後条件)
  = 型レベルで安全な状態管理
```

#### Hybrid 2: Vesperia + RWBY = リアクティブ並列ストリーム

横断的なイベント伝播 (Vesperia) + 並列ワーカーによる処理 (RWBY) = リアクティブストリーム。RxJava, Akka Streams, Project Reactor がこれに該当する。

#### Hybrid 3: Fate + Alchemy = 契約による依存性注入

ポート/アダプタ構造 (Fate) + 各ポートの契約 (Alchemy) = 型安全な DI。Haskell の Tagless Final、Scala の ZIO Layer がこれに該当する。

#### Hybrid 4: ベルカ + 黒執事 = Saga パターン

三項変換 (ベルカ) + 不可逆操作の確実な実行 (黒執事) = Saga パターン。マイクロサービスにおける分散トランザクション処理。

#### Hybrid 5: ミッドチルダ + 夜天 = メタ的な層構造

整然とした階層 (ミッドチルダ) + メタプログラミングによる動的拡張 (夜天) = プラグイン可能なアーキテクチャ。Emacs、IntelliJ Platform、VS Code Extensions がこれに該当する。

### 5.3 未発見の合成領域

既存の人気フレームワークが特定の合成として理解できるなら、**まだ試されていない合成**の中に未発見の優位性が眠っている可能性がある。

例えば:
- うみねこ式 (離散分類) + RWBY式 (フラクタル並列) — まだ広く実装されていない
- 黒執事式 (不可逆契約) + 夜天の書式 (メタ) — リソース管理のメタプログラミング
- Vesperia式 + Alchemy式 — 横断的契約

これらの未探索の組み合わせは、本ツールが**新しいアーキテクチャパターンの発見器**として機能する可能性を示唆する。

### 5.4 合成の図像表現

合成された式は、図像的にも合成として描かれる。具体的には:

- **並列合成**: 隣接配置 + 結合点
- **入れ子合成**: 一方の式の内部に他方の式が描き込まれる
- **重ね合わせ合成**: 透明度を持つ複数の式が同じ位置に重ねられる
- **変換合成**: アニメーションまたは時系列スライダーで式の変化が表現される

特に重ね合わせ合成は、立体ビュー (Phase 6) との親和性が高い。Z 軸方向に異なる式の層が積まれ、視点を変えると合成の構造が浮かび上がる。

## 6. Designer Mode のインターフェース設計

### 6.1 対話的なコマンド

```bash
$ mystical design --interactive

? どんな性質を求めますか? (複数選択可)
  ❯ ◉ fault-tolerance
    ◯ 型安全性
    ◯ 横断的関心事の分離
    ◯ メタプログラミング
    ◯ 並列性
    
? 適用ドメインは?
  ❯ API ハンドラ
    ワーカープロセス
    状態管理
    ドメインロジック
    
推奨される式: RWBY式 (6-fold Divergent fractal)
類似レシピ: fault-tolerance-via-fractal

? 骨格コードを生成しますか?
  ❯ はい (Rust)
    はい (TypeScript)
    いいえ (図像のみ表示)
```

### 6.2 非対話的なコマンド

```bash
# レシピを直接指定
$ mystical design --recipe fault-tolerance-via-fractal --lang rust --output src/api/

# 性質を指定
$ mystical design --property "type safety, immutability" --output sketch/

# 異質適用を試す
$ mystical design --domain state-management --force-style alchemy --output sketch/

# 既存コードを別の式で再設計
$ mystical refactor --from src/state/reducer.rs --to-style alchemy --output suggestion/
```

### 6.3 出力形式

Designer Mode は以下を出力する:

1. **魔法陣の SVG**: 設計のスケッチとしての図像
2. **骨格コード**: 選択された式に基づくテンプレート (関数シグネチャ、型定義、モジュール構造)
3. **設計ノート**: なぜこの式が選ばれたか、どんなトレードオフがあるか
4. **参照実装へのリンク**: 同じパターンを採用した既存プロジェクト

これにより、設計プロセスが**視覚 + コード + 文書**の三位一体として記録される。

## 7. リファクタリング支援としての式変換

### 7.1 式変換 (Style Refactoring) の概念

既存コードの式を、別の式に**変換**する。これはリファクタリングの新しい単位として位置づけられる。

例: 「このモジュールはベルカ式だが、Alchemy式に変換すべき」「この API ハンドラ群をうみねこ式から RWBY式に変換すべき」。

変換は段階的に行われる:

**Phase 1**: 現状の式を Analyzer で診断
**Phase 2**: 目標の式を Designer で決定
**Phase 3**: 両者の差分を可視化
**Phase 4**: 段階的な変換ステップを提案 (一度に全部変えない)
**Phase 5**: 各ステップで Analyzer により進捗を確認

### 7.2 変換の難易度

式の変換難易度は、対称性のオーダーの距離に概ね比例する。

- 同じオーダー内のモード変換 (うみねこ式 ↔ Vesperia式): 比較的容易
- 隣接オーダーの変換 (ベルカ式 ↔ Alchemy式): 中程度
- 離れたオーダーの変換 (ベルカ式 ↔ Fate式): 困難
- 夜天の書式への変換: 最も困難 (メタレベルへの飛躍を伴う)

ツールはこの難易度を提示し、ユーザーが見通しを持って変換に取り組めるよう支援する。

### 7.3 変換の不可逆性

一部の式変換は不可逆である (情報を失う)。例:

- 黒執事式 → Vesperia式: 静的契約の保証が失われる
- Alchemy式 → ベルカ式: 契約情報が失われる
- 夜天の書式 → ミッドチルダ式: メタレベルの構造が失われる

これらの不可逆変換を行う際は、ツールが警告を出す。

### 8.1 Phase との対応

Designer Mode の実装は段階的に進める:

**Phase 1**: Analyzer Mode のみ。Designer 機能は未実装。

## 9. 設計支援ツールとしての歴史的位置づけ

### 9.1 先行する設計支援の系譜

設計支援ツールには長い歴史がある:

- **CASE ツール** (1980s-): UML エディタ、ER 図ツール
- **パターンランゲージ** (1977-): Alexander のパターン、GoF パターン
- **モデル駆動開発** (2000s-): MDA, Sirius
- **ドメイン特化言語** (2000s-): JetBrains MPS, Xtext
- **AI 支援開発** (2020s-): GitHub Copilot, Cursor

本ツールはこれらの系譜に連なるが、**視覚的な対称性に基づく式の選択**という独自のアプローチを取る点で差別化される。

### 9.2 AI との関係

近年の AI 支援開発ツール (Copilot, Cursor, Claude Code) との関係について。

これらのツールは「自然言語からコードを生成する」が、設計の意図や構造的な選択は明示化されない。本ツールの Designer Mode は、**設計の意図を式として明示化する**点で AI 支援とは異なる役割を持つ。

両者は競合ではなく補完関係にある:

```
[人間] → 求める性質を言語化
    ↓
[Mystical Designer] → 性質に対応する式を選択、骨格を生成
    ↓
[AI Code Assistant] → 骨格を埋める詳細コードを生成
    ↓
[Mystical Analyzer] → 生成されたコードが意図通りの式になっているか検証
```

このパイプラインにおいて、Mystical は**設計レベルの意図と実装レベルの現実を繋ぐ装置**として機能する。

## 10. ツールの最終的な位置づけの再定義

これまでの全議論を統合すると、Mystical CI は以下のように位置づけられる:

**目的の二重化**: 
- (a) 既存コードを視覚的に診断する装置 (Analyzer Mode)
- (b) 求める性質からコードを設計する装置 (Designer Mode)

**価値の系譜**: 
- 可視化ツール (CodeCity, Sourcetrail) の延長
- パターン言語 (Alexander, GoF) の進化形
- 設計支援ツール (CASE, MDA) の現代版
- AI 支援開発と補完関係

**独自性**: 
- 視覚的対称性に基づく式の体系
- ファンタジー図像学を借用した記号体系
- 双方向性 (解析と設計の往復)
- 異質式適用による新パラダイムの引き寄せ

**到達点**:
- コードベース全体の景観可視化 (Appendix B)
- 設計レシピ集と式駆動設計 (本文書)
- 立体ビューによる合成の表現 (Phase 6)
- VR/AR 対応の空間 UI (将来構想)

本ツールが「真面目に作れば」、コード可視化ツールの枠を超えて、**ソフトウェア設計の新しい記号体系**として機能する可能性を持つ。それは Christopher Alexander が建築で行ったことの、ソフトウェア版である。
