## §4.3 追補 — EdgeLayerData の再設計 (Phase 3.4 の追認)

v0.2 §4.3 の既定方針どおり、`EdgeLayerData` を Sigil 側 `LayerData` と同じ
`Option<XxxInfo>` 積層構造へ破壊的に再設計した:

```json
{
  "data_flow": { "variables": ["total"] },
  "profile": null,
  "labels": []
}
```

- 旧 `data_volume: Option<f64>` は `data_flow.variables` (流れる変数名の辞書順列挙)
  として正式化。量 (`variables.len()`) だけでなく「何が流れているか」を保持し、
  近似の誤検出をデバッグ可能にする (説明可能性の優先)
- 旧 `call_frequency: Option<u64>` は `profile.call_frequency` (Phase 5+) へ移設
- これにより Sigil / Edge のレイヤー表現が対称になり、v0.2 §4.3 の宿題を解消する

## §5.1 追補 — data_flow レイヤーの具体化 (Phase 3.4)

syn ベースの**意味解決なしの近似**で Use-Def chains を充填する:

- **def**: `let` 束縛・関数引数・再代入 (`x = e` / `x += e`)。シャドーイングは新 def
- **use**: スコープで解決できた単一セグメント識別子の出現
- **追わないもの**: クロージャ本体・マクロ内・非ローカルへの代入 (lhs は use のみ)・
  借用/寿命 (Phase 5+)
- リング単位の集計 (`DataFlowInfo`): `use_def_chains` = そのリングで def され
  1回以上 use された変数の数 / `longest_chain` = チェーンを構成する変数に触れた
  Operation 数 (def 1 + use 回数) の最大値。二重計上を防ぐ規約は ControlFlowInfo と同じ
  (def したリング側にのみ計上)
- **DataFlow Edge**: def したリングと異なるリングで use されたとき
  `(def リング) → (use リング)` の `EdgeKind::DataFlow` Edge を生成する。
  ループ内の再代入は新 def として扱うため、「ループで変換された値が親へ還流する」
  上り方向の Edge も生まれる (ベルカ式 §14.2 の「変換 → 消費」の根拠)
- Operation 単位の `defs` / `uses` (payload) にどの構文から取ったかを残す
- ミッドチルダ式の描画・レイアウト・Spell Diff の木構造は **ControlFlow Edge のみ**を
  見る (DataFlow Edge の描画はベルカ式 §14 の仕事)

## §5.4 追補 — 位置共有制約の適用範囲

v0.2 §5.4 の位置共有制約 (どのレイヤー組み合わせでも同じ位置) は、
**同一の式 (RenderStyle) の内側でのみ**適用されると明確化する。
ミッドチルダ式とベルカ式は同じ IR の別射影 (§14.1) であり、式をまたいだ
Sigil 位置の共有は要求しない (要求すると両式のレイアウト自由度が互いを拘束する)。
v0.2 §5.4 の記述は Phase 2 時点で式が1つしかなかったための省略であり、本追補が優先する。

## §8 追補 — フィルター言語: `highlight:` の解禁 (Phase 3.2)

予約語だった `highlight:` のうち、最小形を Phase 3.2 で導入する。

```
highlight: changed
```

- **解禁する語彙**: `highlight: changed` のみ (PR 差分由来の変更箇所。notes §3.2 の
  `changed_in_pr` はこの `changed` に名称統合し、別語彙としては導入しない)
- **引き続き予約**: `highlight:` のその他の値、および `filter:` (メトリクス条件) 全体。
  予約語の指定は従来どおり明示エラーで案内する。
- `highlight: changed` は **diff 文脈** (`SpellDiff`、§9.2) があるときのみ有効。
  文脈なしで指定された場合は「diff 文脈がありません (magia diff --svg で使用してください)」
  の案内エラーとする (黙って無視しない — v0.2 §8 と同じ設計原則)。
- 強調はレイヤーと独立した**強調チャネル** `<g class="overlay-diff">` として描画され、
  レイヤーの show/hide の影響を受けない。

## §9 正式化 — CI 統合 (Phase 3.1〜3.3)

notes §4 を正式化する。

### §9.1 PR コメント投稿のフロー

PR ごとに以下を実行する:

1. ベースブランチと PR ブランチの両方を解析する
2. 変更された関数だけ図形を生成する (`magia changed --git <REV>` で列挙)
3. 差分の可視情報 (テキスト要約、および強調 SVG への参照または埋め込み) を PR コメントに投稿する (sticky comment 方式で更新。SVG の提示方式は実装時判断)
4. メトリクス変化をテーブルで併記する
5. しきい値超過時のみ status check を fail させる

CI ロジックは `magia` コマンド側に寄せ、ワークフロー定義は薄く保つ
(ローカルで同じ操作が再現できることをデバッグ可能性の要件とする)。

### §9.2 差分エンジンの契約 (Phase 3.1)

- 入力は同一関数の2つの `MagiaGraph` (before / after)。出力 `SpellDiff` は
  **追加 / 削除 / 変更**された Sigil・Edge の集合と、メトリクス変化 (`MetricsDelta`:
  複雑度・副作用カテゴリ数・リング数・召喚記号 (SummonGlyph) 数・早期リターン数の差分) を持つ。
- **`SigilId` はリビジョン間で安定しない**。対応付けは ID でなく**構造マッチング**で行う:
  リングは (kind, `AuxRingRole` の anchor_operation / ordinal / 分岐・ループ種別 —
  v0.2 §4.2 で定義) を、召喚記号は (`call_target`, 所属リング) をキーに突き合わせる。曖昧なケース (同キーが複数) は ordinal /
  ソース出現順の貪欲対応で決定論的に解消し、余りを追加/削除とする。
- 同一入力からは空 diff。diff 自体も決定論的 (同じ入力ペアからは常に同一の SpellDiff)。

### §9.3 しきい値判定の最小主義

v0.1 §2 の設計原則「ツールは判決を下さず、注意を誘導する」と整合させるため、
status check を fail させる条件は**明確に有害な変化のみを限定列挙**する:

- Phase 3.3 時点の fail 対象: **unsafe ブロックの新規追加** のみ。
  notes §4.3 が例示した「複雑度の大幅増加」「ホットパス上の DB クエリ追加」は、
  前者はしきい値の妥当な定数が運用実績なしに決められないため、後者はホットパス情報
  (動的解析、Phase 5) が未導入のため、いずれも初期対象から外す
- その他のメトリクス変化 (複雑度増加・副作用カテゴリ追加等) は警告表示に留める
- fail 対象の拡充は運用実績に基づいて判断する (過剰判定は無視されるツールを生む)

## §14 新設 — 描画様式と解析軸の対応 (Phase 3.5〜)

notes §7 を正式化する。前提だった **Phase 3.4 のデータフロー IR**
(Use-Def chains の充填と `EdgeLayerData` の破壊的再設計) は完了済み —
スキーマは §4.3 追補・§5.1 追補を参照。

### §14.1 二大可視化軸

- **Call Graph / Dependency Graph**: 「誰が誰を呼ぶか」。AST + Symbol Resolution で得られる。
  → **ミッドチルダ式** (ノード中心の同心円構造、Phase 1 で実装済み)
- **Data Flow Graph**: 「値がどこで生まれ、どこを通り、どこで使われるか」。
  Use-Def chains 等のデータフロー解析を要する。
  → **ベルカ式** (エッジ・力場中心の三角構造、Phase 3.4 のデータフロー IR を前提に Phase 3.5 で実装)

両式は排他的選択ではなく、**同じ IR を別の軸で見た補完的なビュー**である。
レイアウト・レンダラは式ごとに持つが、IR は共有する (v0.1 §4.1 の帰結)。

### §14.2 ベルカ式の構成 (Phase 3.5 の契約)

- 関数内の値の流れを「生成 / 変換 / 消費」の3極に集約し、三角形の頂点に配置する
- 頂点の重み (含まれる Operation 数) の偏りが三角形の歪みとして現れる
- **力場**: 各頂点を中心とした放射状グラデーションのアディティブ合成。
  場の濃淡がデータフロー量を表す
- 決定論・乱数なしの規約はミッドチルダ式と共通 (v0.1 §6.1.4)

### §14.3 式の自動推奨 (Phase 3.5 は提案メッセージのみ)

AST の形からの推奨ヒューリスティクスは**提案に留め、自動切替はしない**:

- `(A, B) -> A` 形のシグネチャ (Reducer パターン) を検出したとき、CLI 出力に
  「構造的にはベルカ式が適合します (`--style belka`)」と1行提案する
- 逆方向 (ベルカ式指定だが3点関係が不明瞭) のフォールバック提案は Phase 3 振り返りで判断

### §14.4 スタイル選択のインターフェース

- CLI: `--style <midchilda|belka>` (既定 midchilda)。`RenderStyle` enum は v0.1 §6 で確保済み
- dev-server: クエリ `?style=belka` とパレットのトグルで切替 (位置共有制約 §5.4 は
  **式の内側でのみ**適用される — 式をまたぐ位置の共有は要求しない)

## §16 新設 — 配置済み IR の JSON エクスポート (Phase 4.0.9)

Vue クライアント (Phase 4.0.5〜) が魔法陣を描画するための境界契約。
`magia serve` の `/spell/<fn>` レスポンスの `ir` フィールドとして配信される。

### 設計原則

- **レイアウトは Rust、描画は Vue** (POSD 分担): 座標・半径・操作ドット配置・
  シグネチャ円弧 (衝突回避込み) は全て Rust 側で確定済みの値を返す
- **記号の頂点計算は描画**: 分岐 Y 字・ループ三角・早期リターン矢の形は Vue 側が
  種別 + 向きから組み立てる (形の意匠は描画側の自由度)
- `id` は JSON 内の相互参照 (`edges[].from/to`) 専用。SigilId はパースごとに
  変わりうるため永続化に使わない (§9.2 の ID 非公開方針と同根)
- ミッドチルダ式のみ。ベルカ式は Phase 4.3 の Vue SSR 移行まで SVG 文字列
  (`svg_belka`) を併載する

### スキーマ (実装: `magia-core::render::ir_export`)

```jsonc
{
  "view_box": [minX, minY, width, height],
  "rings": [
    {
      "id": 0,
      "role": "main" | "aux",
      "x": 0.0, "y": 0.0, "radius": 120.0,
      "is_async": false,            // true なら内側に二重線 (§6.1.3)
      "symbol": null | "branch" | "loop",   // リング中央の制御記号 (main は常に null)
      "early_return": null | [ux, uy],      // 早期リターン矢印の向き (単位ベクトル)
      "operations": [               // 3時起点・反時計回りの配置済みドット (§6.1.2)
        { "x": 104.0, "y": 0.0, "radius": 3.5, "effect": "pure" }
      ]
    }
  ],
  "glyphs": [                       // 召喚印 (SummonGlyph / GateGlyph)
    { "id": 7, "x": 162.0, "y": 0.0, "radius": 14.0, "effect": "io" }
  ],
  "edges": [                        // 制御フロー接続。端点はリング表面 (描画側で計算)
    { "from": 0, "to": 7 }
  ],
  "signature": null | {
    "text": "fn ...", "arc_path": "M...",   // 円弧は衝突回避済み (§6.1.3)
    "arc_radius": 86.4,                       // 弧長 (π×半径) フィットの入力 (2026-06-12 追補)
    // 表示組み立て用の部品 (2026-06-12 追補)。クライアントの引数表示オプション
    // (変数名/型名) が ON のとき text の代わりに組み立てる。旧 IR では省略。
    "name": "foo",                            // optional
    "args": [{"name": "a", "ty": "i32"}],     // optional (空なら省略)
    "ret": "Result<(), Error>"                // optional (unit 戻り値は省略)
  },
  "return_branch": null | [x, y]    // Result/Option の正常/異常分岐線の起点 (9時)
}
```

`effect` は §6.1.3 の効果カテゴリ語彙 (`pure / io / network / db / filesystem / unsafe`)。
色への変換は描画側 (uno.config.ts の theme = palette.rs と同語彙) が行う。
