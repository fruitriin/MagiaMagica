# MagiaMagica 仕様書 v0.1

> **本文書の位置づけ**
> 本文書は、プログラミング言語 Mystical の図形表現思想を、コード品質可視化ツールへと拡張する設計仕様である。最終的には多次元的なコード解析・可視化基盤を目指すが、第一段階 (Phase 1) では平面図形1枚の生成に集中する。内部データ構造は将来の立体ビューに耐える設計を最初から確保するが、立体レンダリングの実装は保留する。

---

## 1. プロジェクト概要

### 1.1 ビジョン

MagiaMagica は、ソースコードを魔法陣状の図形に変換することで、コードレビューとリファクタリングにおける**注意誘導**を支援するツールである。静的解析の結果を、人間の視覚野が並列処理可能な空間表現に変換し、レビュアーの認知資源を最適に配分させることを目的とする。

ツールの自己定義としては、品質保証ツールではなく、**注意のスポットライト**である。判決を下すのではなく、目を向けるべき場所を指し示す。判定の権限は常に人間に残す。

### 1.2 思想的背景

ツールはオリジナルプログラミング言語 Mystical (Denis M., suberic.net) の図形思想から着想を得ている。Mystical はリングと記号で命令を表現するアプリオリ言語で、ソースコードを魔導書のように記述する。本ツールはこの図形表現を、既存言語 (Rust, TypeScript 等) のコード可視化に転用する。

ただし、Mystical の構文をそのまま模倣するのではなく、その**設計思想**——図形による構造表現と、空間的なゲシュタルトの活用——を継承し、CI/dev ツールに必要な要素 (インクリメンタル解析、レイヤー切替、差分表示) を独自に追加する。

## 2. 設計原則

本ツール全体を貫く5つの原則を定める。すべての実装判断はこれらの原則に照らして評価される。

**原則 1: 完全な情報を内部に保持する**

中間表現 (IR) は、将来想定される全ての関心軸 (制御フロー、データフロー、副作用、テスト被覆、プロファイル、git 履歴、AI 注釈等) を格納可能な多次元構造とする。Phase 1 で実際に値を埋めるフィールドは限定されるが、スキーマ上の場所は最初から確保する。これにより、Phase が進むごとにスキーマ破壊を起こさず機能を追加できる。

**原則 2: デフォルトは最も単純な射影を提示する**

ユーザーが何も設定しなければ、最小限のレイヤー (典型的には制御フローと外部呼び出し) のみを描画する。情報を増やすのは常にユーザーの明示的な操作を要する。

**原則 3: 射影は明示的に拡張・縮小できる**

ユーザーはレイヤーの追加・削除・透明度調整・色相変更を、対話的に行える。射影パラメータは保存・共有・バージョン管理可能なテキスト形式 (フィルター言語) で表現される。

**原則 4: 複雑な射影は明示的な呼び出しを要求する**

立体ビューや全レイヤー同時表示など、認知負荷が高い表現は、明示的なコマンドや操作を経て初めて表示される。「うっかり開いてしまう」ことがないようにする。

**原則 5: 任意の射影は保存・共有・バージョン管理できる**

「セキュリティレビュー断面」「アーキテクチャ監査断面」のようなプリセットを、チーム内で git 共有できる。フィルター定義は実行可能なテキストファイルとして扱う。

## 3. アーキテクチャ

### 3.1 全体構成

```
[Source Code]
     ↓
[Language Adapter]  ←─ 言語ごとに分離
     ↓
[MagiaIR]       ←─ 言語非依存・多次元
     ↓
[Analysis Engine]   ←─ メトリクス算出
     ↓
[Projection Layer]  ←─ フィルター適用
     ↓
[Layout Engine]     ←─ 位置決定 (安定レイアウト)
     ↓
[Render Engine]     ←─ SVG 生成
     ↓
[Output]            ←─ ローカル表示 / CI 出力
```

各層は IR との境界を明確に持ち、層の入れ替えが可能な構造とする。

### 3.2 モジュール構成

中核モジュール (Phase 1 のスコープ):

`magia-core`
言語非依存の IR 定義、解析エンジン、レイアウトエンジン、平面 SVG レンダラ、フィルター適用ロジックを含む。Rust で実装する。

`magia-rust`
Rust ソースを IR に変換するアダプタ。`syn` および `rust-analyzer` のセマンティック情報を利用する。Phase 1 における唯一の言語アダプタ。

`magia-cli`
ローカル実行用のコマンドラインインターフェース。Phase 1 では静止画 SVG 出力のみを提供。

将来の拡張モジュール (Phase 1 では実装しない):

- `magia-watch` (dev-server)
- `magia-web` (TypeScript/React アダプタ)
- `magia-node` (Node.js バックエンドアダプタ)
- `magia-ci` (GitHub Actions 等の CI 統合)
- `magia-server` (ダッシュボード)
- `magia-3d` (立体ビューレンダラ)

### 3.3 実装言語

中核は Rust で実装する。理由は、大規模モノレポでの解析速度、Rust アダプタとの同言語実装による親和性、将来的な WASM コンパイルによる Web 配信の容易性、および型安全性による IR の整合性保証である。

## 4. 中間表現 (IR) スキーマ

### 4.1 設計方針

IR は将来の全機能を見越した完全構造として設計する。Phase 1 で値が入らないフィールドは `Option` または空コレクションを許容するが、スキーマ自体は最初から定義する。

### 4.2 主要型定義 (擬似 Rust 構文)

```rust
/// プロジェクト全体のグラフ
struct MagiaGraph {
    modules: Vec<Module>,
    cross_module_edges: Vec<Edge>,
    metadata: ProjectMetadata,
}

/// モジュール (ファイル / クレート / コンポーネント等)
struct Module {
    id: ModuleId,
    name: String,
    sigils: Vec<Sigil>,
    edges: Vec<Edge>,
}

/// 図形ノード = リング・記号の単位
struct Sigil {
    id: SigilId,
    kind: SigilKind,
    content: Vec<Operation>,
    layers: LayerData,           // 多次元情報を保持
    source_location: SourceSpan,
    cardinality: Cardinality,
}

enum SigilKind {
    MainRing,      // 関数本体
    AuxRing,       // 補助ロジック / 内部ブロック
    SummonGlyph,   // 外部呼び出し
    GateGlyph,     // エントリポイント / 公開境界
}

/// リング内の処理単位
struct Operation {
    kind: OperationKind,
    effects: EffectSet,
    payload: OperationPayload,
}

enum OperationKind {
    Compute,
    Branch,
    Loop,
    Match,
    Call,
    Await,
    Yield,
    Return,
    Throw,
}

/// 副作用カテゴリ (将来の拡張余地を持つ)
struct EffectSet {
    pure: bool,
    io: bool,
    network: bool,
    db: bool,
    filesystem: bool,
    mutation: bool,
    unsafe_block: bool,
    custom: Vec<CustomEffect>,
}

/// 線 (リング間・記号間の接続)
struct Edge {
    source: SigilId,
    target: SigilId,
    kind: EdgeKind,
    cardinality: f64,   // 接点の太さ・情報量
    layers: EdgeLayerData,
}

enum EdgeKind {
    ControlFlow,
    DataFlow,
    Dependency,
    Inheritance,
    Implementation,
}

/// 多次元レイヤー情報 (Phase 1 では一部のみ埋まる)
struct LayerData {
    control_flow: Option<ControlFlowInfo>,
    data_flow: Option<DataFlowInfo>,
    type_info: Option<TypeInfo>,
    lifetime: Option<LifetimeInfo>,
    concurrency: Option<ConcurrencyInfo>,
    test_coverage: Option<CoverageInfo>,   // Phase 1: None
    profile: Option<ProfileInfo>,           // Phase 1: None
    git_churn: Option<ChurnInfo>,           // Phase 1: None
    security: Option<SecurityInfo>,         // Phase 1: None
    ai_annotations: Vec<AiAnnotation>,      // Phase 1: empty
}
```

### 4.3 立体格納可能性の確保

`LayerData` および `EdgeLayerData` を **直交する独立フィールドの集合**として設計することで、Z 軸方向 (関心の積層) の表現が後から自然に追加できる構造を保証する。Phase 1 では大半のフィールドが空だが、フィールドの存在自体が将来の Z 軸スタックの基礎となる。

将来の立体ビューは、`LayerData` の各フィールドを Z 軸上の異なる平面にマッピングする射影として実装される。Phase 1 ではこの射影を「単一レイヤーへの選択」として平面化する。

## 5. レイヤー定義 (関心の軸)

### 5.1 想定される全レイヤーのカタログ

将来的に実装が想定される関心レイヤーを以下に列挙する。Phase 1 ではこのうち最小限のみを実装するが、IR には全てのレイヤー用フィールドを確保する。

`control_flow` — 制御フロー (分岐、ループ、例外経路)
`data_flow` — データの流れ、変数の伝播
`effects` — 副作用カテゴリ (IO/Network/DB/Filesystem/Unsafe)
`type_info` — 型注釈、推論結果、ジェネリクス
`lifetime` — Rust の借用・寿命情報
`concurrency` — async/await、channel、mutex、atomic
`test_coverage` — テスト被覆率
`profile` — 実行頻度、ホットパス
`git_churn` — 変更頻度、変更履歴
`security` — 静的解析の警告
`dependency_direction` — アーキテクチャ層 (UI/Domain/Infra) の越境
`dynamic_dependency` — 実行時にトレースした依存
`ai_annotations` — LLM ベース解析の指摘

### 5.2 Phase 1 で実装するレイヤー

Phase 1 では以下のみを実装する。残りはスキーマ上のフィールドのみ確保し、値は埋めない。

`control_flow` — Rust の if / else / match / loop / while / for / `?` 演算子による分岐
`effects` — 関数の効果カテゴリ判定 (call site の crate 名と関数名からの推定)
`type_info` — 関数シグネチャ (引数型、戻り値型) の基本情報のみ

これら以外のレイヤーは Phase 2 以降で段階的に追加する。

## 6. レンダリングと視覚化

### 6.1 平面レンダリング (Phase 1 で実装)

#### 6.1.1 描画単位

Phase 1 の描画単位は **関数1つ** とする。1つの SVG ファイルは1つの関数を表現する。複数関数を1枚に描く機能は Phase 1 のスコープ外とする。

#### 6.1.2 基本構造

関数本体を中心の **メインリング** として描画する。メインリングは円形で、内部に処理を反時計回りに配置する。Mystical 原典に従い、開始点はメインリングの3時の位置とし、反時計回りに流れる。

関数内の主要な制御構造ブロック (if/else の分岐先、ループ本体、match の各アーム) を **補助リング** として、メインリングの外周に配置し、メインリングと線で接続する。

外部関数呼び出しは **召喚記号** として、メインリングまたは関連する補助リングから線を伸ばした先に配置する。

#### 6.1.3 記号体系

Phase 1 で実装する記号セットを以下に定める。

制御構造:
- 分岐 (if/match): 二股以上に分かれる線の合流点
- ループ (for/while/loop): 円形の補助リング (内側に小さい矢印)
- 早期リターン (return/`?`): リングの内側から外へ抜ける矢印

効果カテゴリ (色相で区別):
- 純粋計算: 黒
- IO: 青
- ネットワーク: 紫
- DB: 緑
- ファイルシステム: 茶
- Unsafe: 赤 (Rust のみ)

その他:
- 関数シグネチャ: メインリングの最外周に円弧ラベル
- async fn: メインリングの輪郭を二重線にする
- Result/Option 戻り値: メインリングから出る分岐線 (正常パスと異常パス)

#### 6.1.4 安定レイアウト要件

レイアウトは決定論的であり、同じ IR からは常に同一の SVG を生成する。乱数を用いる場合は固定シードを使用する。

レイアウトアルゴリズムは以下の優先順位で位置を決定する:

1. メインリングを画面中央に配置 (常に同じ位置)
2. 補助リングを、関連するメインリング上の制御フロー位置に基づいて極座標で配置
3. 召喚記号を、最も関連の深いリングから一定距離の位置に配置
4. 線の交差を最小化する局所最適化を最後に適用

この順序により、関数の局所変更が図形の局所変更にとどまることを保証する。Phase 2 以降の差分表示の前提条件となる。

#### 6.1.5 出力形式

SVG ファイルを標準出力または指定パスに書き出す。SVG 内部の構造は、レイヤーごとに `<g>` 要素を分離し、`class` 属性でレイヤー名を付与する。これにより将来の対話的レイヤー切替で、SVG を再生成せず CSS のみで切替可能とする。

### 6.2 立体ビューに関する考察 (実装は保留)

#### 6.2.1 課題設定

平面の SVG 1枚で表現できる情報量には認知的な限界がある。複数のレイヤーを同時表示すると視覚的に混雑し、注意誘導という本ツールの目的を損なう。一方で、関心レイヤー間の**相関**——例えば「副作用が多く、かつテスト被覆が低く、かつ最近変更頻度が高い領域」のような複合条件——は、複数レイヤーを横断的に見る必要がある。

立体ビューは、各レイヤーを Z 軸上の異なる平面に配置し、視点を回転させることで複数レイヤーの相関を**柱**として浮かび上がらせる表現手法である。これは平面では原理的に表現できない情報密度を実現する。

#### 6.2.2 価値

立体ビューが正当に機能するユースケースは以下を想定する。

四半期に一度のアーキテクチャ監査、大規模リファクタリング前後の構造比較、新規参画者へのコードベース概観の教示、コード基盤の異常検知 (立体的な突起の検出)、カンファレンスや経営報告での視覚資料。

これらはいずれも頻度は低いが、必要時には平面表現では代替不可能な価値を提供する。

#### 6.2.3 思想的位置づけ

立体ビューはツールの**到達天井**を決める機能であり、デフォルトでは表示されず、明示的な呼び出しによってのみ起動される。ファンタジー文学において立体魔法陣が特殊な状況・権限・代償を要する儀式として描かれるのと同様に、立体ビューは認知的代償と引き換えに通常では到達不可能な情報密度を提供する位置に置かれる。

存在しない機能と、存在するが普段は使わない機能は、ツールの設計上は全く異なるレイヤーに属する。前者はツールの限界を決め、後者はツールの自由度を決める。本ツールは後者を選択する。

#### 6.2.4 Phase 1 における対応

Phase 1 では立体ビューの**実装**は行わない。しかし、IR が立体表現に耐える多次元構造を最初から保持する点で、立体ビューへの拡張性は仕様レベルで保証される。

Phase 1 完了時点で、`LayerData` の各フィールドは独立した平面射影として個別に描画可能であり、それらを将来 Z 軸方向に積層する実装を追加するだけで立体ビューが成立する設計となっている。

## 7. dev-server とレイアウト (Phase 2 以降)

dev-server 機能、ファイル監視、インクリメンタル再計算、WebSocket ストリーミング、対話的レイヤー切替は Phase 2 以降で実装する。これらは Phase 1 の静止画 SVG 生成を基礎として、その上に対話層を追加する形で実装される。

Phase 1 の SVG 出力は、レイヤーを `<g>` 要素として分離した構造を持つため、Phase 2 で SVG を再生成せず CSS class の切替だけで対話的レイヤー操作を実装できる準備が整っている。

## 8. フィルター言語 (Phase 2 以降)

レイヤーの組み合わせ、強調条件、非表示条件を記述する DSL を提供する予定だが、Phase 1 では実装しない。Phase 1 では CLI 引数で表示するレイヤーを指定する単純なインターフェースのみ提供する。

将来的なフィルター言語の構文例:

```
show: control_flow + effects[network, db]
highlight: changed_in_pr
hide: test_covered > 0.8
filter: complexity > 7
```

このフィルター定義はテキストファイルとしてプロジェクトに保存し、git でバージョン管理できる。

## 9. CI 統合 (Phase 3 以降)

GitHub Actions, GitLab CI 等への統合、PR コメントへの自動投稿、メトリクスのしきい値判定によるビルド失敗は Phase 3 以降で実装する。Phase 1 はローカル CLI のみ。

## 10. Phase 1 スコープの厳密定義

### 10.1 実装する機能

対象言語: Rust のみ
対象プロジェクト形式: 単一の Rust ソースファイル、または `cargo` で管理される単一クレート
描画単位: 関数1つにつき SVG 1枚
出力形式: SVG ファイル (静止画)
収集する情報: コードから直接読み取れる依存グラフのみ (具体的には 10.3 で列挙)
実装するレイヤー: control_flow, effects, type_info の3つ
レンダリング: 平面のみ
実行形態: ローカル CLI、単発実行
言語アダプタ: `magia-rust` のみ
中核実装: `magia-core` の Rust 実装

### 10.2 実装しない機能

dev-server / ファイル監視 / インクリメンタル更新
対話的レイヤー切替
立体ビューのレンダリング
複数関数を1枚に描く機能
差分表示 (PR 前後の比較)
CI 統合 / PR コメント投稿
フィルター DSL
GUI / Web UI
他言語アダプタ (TypeScript, Node 等)
動的解析 / 実行時トレース
テスト被覆統合
プロファイル統合
git 履歴統合
AI 注釈統合
セキュリティ警告統合

### 10.3 Phase 1 で収集する情報の厳密リスト

`magia-rust` が `syn` / `rust-analyzer` を介して収集する情報は以下に限定する。

- 関数定義 (名前、シグネチャ、本体の AST)
- 関数内の制御構造 (if / else / match / loop / while / for)
- 関数内の早期リターン (`return`, `?` 演算子)
- 関数本体からの他関数呼び出し (call site)
- 呼び出し先関数のフルパス (crate::module::function)
- async fn / await 点の位置
- `unsafe` ブロックの有無と範囲
- Result / Option 戻り値型の判定

以下の情報は Phase 1 では収集しない。

- マクロ展開後の構造
- トレイト境界の詳細
- lifetime パラメータ
- generic の具体化情報
- dyn Trait 経由の動的ディスパッチ
- ファイル外の依存関係の意味的解決

## 11. 開発ロードマップ

**Phase 1** (本仕様書のスコープ)
単一 Rust 関数を SVG 1枚の平面魔法陣として出力する CLI。

**Phase 2**
dev-server 化、ファイル監視、WebSocket による対話的レイヤー切替、フィルター DSL の初期実装。

**Phase 3**
CI 統合。PR ベースの差分表示、GitHub Actions 対応、メトリクスしきい値判定。

**Phase 4**
他言語アダプタ。Node.js バックエンド、Web フロントエンド (React/Vue/Svelte) への対応。

**Phase 5**
動的解析の合流。テスト被覆、プロファイル、git churn の統合。AI 注釈レイヤー。

**Phase 6**
立体ビュー実装。Three.js / WebGL による Z 軸積層レンダリング。VR/AR 対応の検討。

各 Phase は独立に実用価値を持つように区切られている。Phase 1 完了時点で「Rust 関数を魔法陣で見られる CLI」として配布可能であり、Phase 2 以降は段階的な拡張となる。

## 12. 残された課題と保留事項

### 12.1 立体レンダリングの設計判断

Z 軸の意味付け (レイヤー積層 / 抽象度 / 時間 / 複合)、視点コントロール、操作インターフェース、認知負荷管理について、Phase 6 で改めて設計する。Phase 1〜5 を通じて蓄積される運用知見が、Phase 6 の設計判断の基礎となる。

### 12.2 動的依存の解決

Node の `require()` 動的解決、TypeScript の `as any` キャスト経由の依存、Rust の `dyn Trait` 動的ディスパッチなどは静的解析では取りきれない。Phase 5 でテスト実行時のトレーサ統合により補完する想定だが、具体的な実装手段は未確定。

### 12.3 ドメインスメル

命名の悪さ、ビジネスロジックの誤り、セキュリティ上の懸念は構造解析では検出できない。LLM ベースの補助解析を別チャネルとして併用する想定だが、Phase 5 で具体化する。

### 12.4 アクセシビリティ

図形ベース表現の視覚障害者対応として、同じ IR から構造化テキストの「呪文書き起こし」を生成する機能が必要だが、Phase 1 のスコープ外。Phase 2 以降で順次対応する。

### 12.5 記号語彙の継続拡張

HTTP, gRPC, AWS, GCP 等のドメイン特化記号は、コミュニティで継続的に拡張される語彙集 (`magia-glyph-*`) として整備する想定。Phase 4 以降に検討する。

## 13. 用語集

**MagiaIR**: 言語非依存の中間表現。多次元のレイヤー情報を保持する。

**Sigil (シジル)**: 図形ノード。リング・記号などの単位。

**Ring (リング)**: 円形の Sigil。関数本体や制御構造を表現する。

**MainRing**: 関数本体を表すリング。図の中心に配置される。

**AuxRing**: 補助リング。制御構造ブロックを表現する。

**Summon Glyph (召喚記号)**: 外部関数呼び出しを表す記号。

**Gate Glyph (門記号)**: エントリポイント、公開境界を表す記号。

**Layer (レイヤー)**: 関心の軸。制御フロー、副作用、テスト被覆等の独立した観点。

**Projection (射影)**: IR から特定レイヤーを選んで生成されるビュー。

**Filter DSL**: レイヤーの組み合わせを記述するドメイン特化言語 (Phase 2 以降)。

**Sigil Cardinality**: リングや記号の重み・情報量。直径や太さに反映される。
