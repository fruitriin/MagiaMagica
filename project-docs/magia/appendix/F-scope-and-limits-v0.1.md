# MagiaMagica Appendix F: ツールの射程と限界 v0.1

> **本文書の位置づけ**
> 本ツールが**何を扱えるか**と**何を扱えないか**を厳密に区別し、明文化する文書である。これまでの Appendix で機能と価値を積み上げてきたが、それらが過剰な期待を生まないよう、ツールの射程を明示的に画定する必要がある。
>
> 本文書はツールの**自己定義**であり、設計判断の最終的な参照点となる。「これはツールでやるべきか、人間がやるべきか、AI に委ねるべきか」という問いに、本文書が答えを提供する。

---

## 1. 形式と意味の区別

### 1.1 二つの領域

本ツールは原理的に**形式 (Form)** の領域のみを扱い、**意味 (Meaning)** の領域には立ち入らない。この区別は本ツールの能力の境界を画定する最も重要な原則である。

**形式の領域**:
- グラフ構造、トポロジー、対称性
- 記号の配置と関係
- 接続パターン、結合点の位置
- 数値メトリクス、統計的特性
- 視覚的な整合性、歪み、密度

**意味の領域**:
- 名前が表す概念の適切さ
- ドメインロジックの正しさ
- ビジネス要件との一致
- ユーザーへの価値
- 文化的・倫理的妥当性

この区別は記号論における Saussure の対比に対応する:

- **Signifier (記号表現)** = 形式の領域 = 本ツールの射程
- **Signified (記号内容)** = 意味の領域 = 本ツールの射程外

シジル (記号) を「描く」「配置する」「接続する」は形式の操作だが、「シジルが何を意味するか」「その意味が適切か」は意味の領域に属する。

### 1.2 なぜこの区別が決定的か

機械が解析できるのは形式である。コンパイラ、リンタ、静的解析ツール、IDE のリファクタリング機能——これらは全て形式の領域で動作する。AST を構築し、グラフを辿り、パターンを照合する。これらの操作には**意味の理解は不要**で、純粋に構文的・構造的な処理として実行可能である。

意味の理解には、ドメイン知識、文脈の把握、価値判断、創造性が必要である。これらは現時点では LLM ですら不完全にしか扱えず、本質的には人間の判断を要求する領域である。

本ツールは形式の領域を徹底的に扱うことで価値を提供する。意味の領域に踏み込もうとすると、ツールは確実に誤った判断を下し、ユーザーの信頼を失う。射程を厳密に守ることが、ツールの長期的な有用性を保証する。

## 2. 見えるものリスト (ツールの射程内)

本ツールが原理的に検出・可視化できる項目を列挙する。これらは全て**形式的に**観察可能である。

### 2.1 構造的配置の異常

- 依存グラフの奇妙な配置 (例: ある関数だけが孤立している、または過剰に中央化している)
- リングの大きさが極端に不均衡 (神オブジェクト、空っぽの関数)
- 補助リングの数が異常 (1つもない、または100個ある)
- 召喚記号 (外部呼び出し) の方向の偏り

### 2.2 結合点 (Junction Glyph) の歪み

- 異なる式のモジュール間の接続が直接的すぎる (Anti-Corruption Layer の不在)
- 結合点が片側だけに偏る (leaky abstraction)
- 結合点の太さの異常 (細すぎる = 情報量不足、太すぎる = 過剰結合)
- 結合点が複数の責務を兼ねている (歪んだキメラ)

### 2.3 参照の散らばり

- 同じドメインの関数群が地理的に散らばっている (式の局所性違反)
- 一つの機能のために多数のモジュールを呼び出す (散弾銃手術)
- 「Feature Envy」: 自モジュール内より他モジュールへの呼び出しが多い (重心オフセット)
- データ依存が予期しない方向に広がっている

### 2.4 循環参照の不格好さ

- モジュール A → B → C → A の閉ループ
- これは図像的に「閉じた円」として視覚化され、明らかな違和感を生む
- 関数レベルの相互再帰は許容されるが、モジュールレベルの循環は通常設計の問題
- 規模が大きい循環ほど深刻 (3モジュール循環より10モジュール循環の方が悪い)

### 2.5 リング・線・記号の不整合

- 矢印の方向が混在 (Clean Architecture では内向きのみ許容、これに反する)
- 同じ機能を表す記号が複数の形式で描かれている (記号体系の不統一)
- 線の太さが情報量と対応していない (細い線で大量のデータが流れる)
- 装飾と機能の混同 (Vesperia 式以外で過剰装飾)

### 2.6 式とコードのミスマッチ

- ベルカ式で描こうとしたが、3点関係が成立していない (Reducer のつもりが、実は手続き型ロジック)
- うみねこ式で描こうとしたが、4分類が網羅的でない
- Profundus 式を狙ったが、インターフェースが肥大化している (Shallow Module)
- Clean Architecture を採用しているが、Dependency Rule に違反している

### 2.7 景観としての違和感

- 全体の式の一貫性が低い (モジュールごとに式がバラバラ)
- 結合トポロジーが想定外 (網状になりすぎ、ハブが集中しすぎ)
- ズームレベル間の連続性が崩れている (全体は Clean Architecture だが、モジュール内部は崩壊)
- 時間軸での景観劣化 (リファクタリングのつもりが式の歪みが増大)

### 2.8 メトリクスの異常

- 深さスコア (Profundus 系)、対称性スコア、結合点健全性、式の一貫性などの数値的異常
- これらは閾値ベースで自動検出可能
- 異常値の検出は形式の領域に属する

## 3. 見えないものリスト (ツールの射程外)

本ツールが原理的に検出できない、または検出すべきでない項目を列挙する。これらは全て**意味の領域**に属する。

### 3.1 命名の良し悪し

最も重要な「見えないもの」。

- 関数名、変数名、モジュール名、型名が適切か
- 名前がドメインを正しく表現しているか
- Ubiquitous Language (DDD 用語) に従っているか
- 略語の使い方が適切か
- 命名規約への適合 (大文字小文字、ハイフンの使用など) は形式の領域だが、**名前が表す概念の適切さ**は意味の領域

「シジル (記号) は形式的に観察できるが、その意味の検査は機械的操作の外にある」というユーザーの観察は、ツールの本質的な限界を端的に表している。

### 3.2 ドメインロジックの正しさ

- ビジネスルールが正しく実装されているか
- 金額計算、税率、割引ロジックなどの妥当性
- ドメイン固有の不変条件 (例: 「注文の合計金額は商品単価×数量+送料」) が守られているか
- アルゴリズムが要件を満たしているか

これらは形式的には正しいコードでも、意味的に間違っていることがある。本ツールはここを判定できない。

### 3.3 ビジネス要件との一致

- 仕様書通りに実装されているか
- 顧客の真の要求と一致しているか (顧客自身も明確化できていない場合が多い)
- 競合との差別化要素が実装されているか
- マーケティング戦略との整合性

これらは技術領域を超えた判断を要求する。

### 3.4 性能特性

- 実行時間、レイテンシー、スループット
- メモリ使用量、CPU 使用率
- スケーラビリティ
- リアルタイム性

形式構造として「ホットパス」を Phase 5 のプロファイル統合で可視化することは可能だが、**実際の性能が要件を満たすか**の判断は別問題。本ツールは「ここが遅い可能性がある」とは言えても「ここが遅い」「ここが要件を満たさない」とは言えない。

### 3.5 セキュリティ脆弱性

- SQL インジェクション、XSS、CSRF などの具体的な脆弱性
- 認証・認可ロジックの妥当性
- 暗号化の適切な使用
- 機密情報の取り扱い

セキュリティ静的解析ツール (Snyk, SonarQube Security 等) は別途存在する。本ツールはこれらと組み合わせて使うものであり、置き換えるものではない。

### 3.6 メモリ効率とリソース管理

- メモリリーク
- ファイルディスクリプタの枯渇
- データベースコネクションの解放漏れ
- 並行性のデッドロック・ライブロック

黒執事式 (不可逆契約) の可視化により**構造的な懸念**を示すことはできるが、実際にメモリリークが起きるかの判定は実行時の動的解析を要する。

### 3.7 ライセンスと法的制約

- 使用しているライブラリのライセンス互換性
- 著作権、特許への抵触
- 業界規制への準拠 (GDPR, HIPAA など)
- 輸出規制

これらは法務領域の話であり、形式コードの構造とは無関係。

### 3.8 ユーザビリティとアクセシビリティの意味

- UI の使いやすさ
- スクリーンリーダー対応の**意味的な**質 (alt 属性の存在は形式、その内容の妥当性は意味)
- 文化的に適切な表現
- 多言語対応の品質

WCAG (アクセシビリティガイドライン) の構造的チェックは形式の領域だが、コンテンツの意味的適切さは別。

### 3.9 美的判断 (意味としての)

形式としての「対称性」「整然」は本ツールが扱う。しかし「これは本当に美しいか」「ユーザーが愛着を持つか」という意味としての美的判断は、形式の整合性とは別の問題である。

### 3.10 倫理的・社会的影響

- AI バイアス、差別的な実装
- プライバシーへの影響
- 環境負荷 (グリーンソフトウェア)
- 社会的影響 (デザイン倫理)

これらは本ツールの射程をはるかに超える。

## 4. 境界の例示: 同じ問題の見え方の違い

同じ問題が、形式的にどう見え、意味的に何を必要とするかを例示する。

### 例 1: 「悪いコード」の検出

```python
def calculate(x, y):
    # 1000 行の処理
    return result
```

**本ツールが見える**:
- メインリングが異常に大きい (1000 行)
- 補助リングが多数 (制御構造の数)
- 関数名と引数名が情報量を持たない (これは形式の話: 名前の「短さ」「単一文字性」)

**本ツールが見えない**:
- `calculate` という名前が**何を**計算しているか
- `x` と `y` が**何を**表しているか
- ロジックが正しいか

### 例 2: 循環依存

```rust
// module_a.rs
use crate::module_b::do_something;

// module_b.rs
use crate::module_a::other_thing;
```

**本ツールが見える**:
- モジュール間の循環参照 (環状結合トポロジー)
- 図像的に閉じたループとして警告色で表示

**本ツールが見えない**:
- この循環依存が**ドメイン的に**意味があるか (稀にあり得る)
- 解消すべきか保つべきか

### 例 3: Deep Module の評価

```typescript
class TokenManager {
  generate(payload: object, expiresIn: number): string { /* 100 行 */ }
  verify(token: string): object { /* 50 行 */ }
  revoke(token: string): void { /* 30 行 */ }
}
```

**本ツールが見える**:
- 公開メソッド 3 つ、実装行数 180 行
- 深さスコア = 180 / 3 = 60 (高い、Deep Module の理想形)
- 引数の数も少ない (Profundus 式に適合)

**本ツールが見えない**:
- このトークン管理が**セキュアか**
- アルゴリズム (JWT? カスタム?) が適切か
- `revoke` の実装がブラックリスト方式か、トークンローテーションか
- ドメインロジックの正しさ

### 例 4: パフォーマンス問題

```python
for item in items:
    db.query(f"SELECT * FROM table WHERE id = {item.id}")
```

**本ツールが見える**:
- ループ内に DB クエリの召喚記号 (副作用層)
- Phase 5 のプロファイル統合で「ホットパス」として強調可能

**本ツールが見えない**:
- これが N+1 クエリ問題であること (意味的な問題)
- 実際にこのコードが遅いか (実行時データが必要)
- どう書き直すべきか (ドメイン知識が必要)

ただし、Phase 5 のプロファイル統合と、Phase 6 の AI 注釈チャネルにより、間接的に補完される可能性はある。

## 5. 「見えないもの」を補う方法

本ツールの射程外の項目を、どうサポートするか。これは「他のツールとの併用」と「人間の判断」の組み合わせで実現する。

### 5.1 専用ツールとの併用

| 見えない項目 | 補完ツール |
|---|---|
| セキュリティ脆弱性 | Snyk, SonarQube Security, OWASP ZAP |
| 性能特性 | APM (Datadog, New Relic), プロファイラ |
| メモリリーク | Valgrind, heap profiler |
| ライセンス | FOSSA, WhiteSource |
| 命名の品質 | LLM (Claude, GPT-4 などの補助) |
| ビジネス要件 | 仕様書、テストケース、ステークホルダーレビュー |
| 倫理的影響 | 倫理委員会、設計レビュー |

これらは本ツールと**直交する**ツール群であり、競合しない。各ツールはそれぞれの領域で価値を発揮する。

### 5.2 AI 注釈チャネル (Phase 6)

LLM ベースの解析を本ツールの**並列レイヤー**として統合する。形式の解析は本ツールが行い、意味の評価は AI 注釈レイヤーが行う。両者の結果を重ね合わせることで、形式と意味の両方が見えるようになる。

これは Phase 6 のロードマップに含まれ、Appendix C v0.2 §6.6 で議論した「AI 時代の新しい統合」の具体形となる。

### 5.3 人間のレビュー

ツールが浮かび上がらせた**注意ポイント**に対し、人間が意味的判断を下す。これが本ツールの基本的な使い方であり、Phase 1 仕様書 §1.1 の「ツールは判決を下さず、注意を誘導する」の本質である。

ツールは「ここに何かがある」と指し示し、人間は「これは問題か、問題でないか」を判断する。この役割分担を明確に保つことが、ツールの信頼性を担保する。

### 5.4 テスト

ドメインロジックの正しさは、最終的にはテストでしか保証できない。本ツールはテスト被覆 (Phase 5) を視覚化できるが、**テストが正しいか**を判定することはできない (テストもコードであり、テストのテストはどうするか、という無限後退に陥る)。

ドメインの正しさは、結局のところ**仕様書、テスト、人間のレビュー、ユーザーフィードバック**の総合で担保される。

## 6. 「形式の徹底」が生む価値

本ツールが意味の領域に踏み込まないことは、機能の限定ではなく、**徹底することで生まれる価値**である。

### 6.1 信頼性

形式の領域に限定することで、ツールの判定は**機械的に再現可能**になる。同じコードに対して、いつ実行しても同じ結果が出る。これは信頼性の基盤である。

意味の領域に踏み込もうとすると、判定が文脈依存になり、ツールの出力にユーザーが懐疑を持つようになる。

### 6.2 高速性

形式の解析は構文解析と統計処理で済み、極めて高速に実行できる。dev-server として常駐し、リアルタイムフィードバックが可能になる。

意味の解析 (LLM 経由) は時間がかかり、開発フローを中断する。これらは別チャネルで非同期に行うべきである。

### 6.3 普遍性

形式の規則は言語・ドメイン・文化を超えて普遍的である。対称性は世界中で対称性であり、循環は世界中で循環である。

意味の規則は文化的・組織的に変動する。「適切な命名」は会社ごとに違う。本ツールが意味に踏み込まないことで、グローバルに使えるツールであり続ける。

### 6.4 教育的価値

形式の規則は学習可能である。新人開発者が「対称性が崩れている」「結合が散らばっている」を視覚的に学ぶことで、徐々に意味の判断もできるようになる (形式から意味への学習転移)。

意味を直接教えるツールは作れないが、形式を通じて意味への気づきを促すことはできる。

### 6.5 哲学的純粋さ

ツールが自分の領域を明確に守ることは、それ自体が美徳である。「すべてをやろうとするツール」は何もうまくできない。「一つのことを徹底するツール」は、その領域で最高の価値を提供する。

Unix 哲学の「Do one thing and do it well」に通じる設計判断である。

## 7. ツールの自己定義の更新

これまでの議論を踏まえて、本ツールの自己定義を更新する。

### 7.1 旧定義 (Phase 1 仕様書より)

> 本ツールは品質保証ツールではなく**注意のスポットライト**である。判決を下すのではなく、目を向けるべき場所を指し示す。判定の権限は常に人間に残す。

### 7.2 新定義 (Appendix F v0.1)

> 本ツールは**コードの形式的構造を視覚化し、形式的異常を注意誘導する装置**である。意味の領域には踏み込まず、専用ツール・AI・人間と補完関係で機能する。
> 
> ツールは「これは何か」「これはこう見える」を提示する。「これは正しいか」「これはどう意味があるか」は問わない。後者は別のチャネル (専用ツール、LLM、人間) に委ねる。
> 
> この限定こそが、ツールの徹底性、信頼性、高速性、普遍性、教育的価値、哲学的純粋さを担保する。

### 7.3 ユーザーへの含意

ユーザーは本ツールを「形式の異常検出器」として使う。意味の判断には別のツール・自分の判断・チームレビュー・ユーザーフィードバックを併用する。

ツールに過剰な期待をせず、ツールの強みを活かす。これが本ツールの正しい使い方である。

## 8. 「形式が示唆する意味」の境界事例

形式と意味は厳密に分けられるが、実際には**形式が意味を示唆する**境界事例が存在する。これらは興味深い分析対象になる。

### 8.1 命名規約と意味

命名規約 (例: getter は `get` で始める、Booleanは `is_` で始める) は形式の領域だが、これに従うことで意味の理解が容易になる。

本ツールは「命名規約に従っているか」は検出できる (形式)。しかし「`get_user` という名前が、本当にユーザーを取得するか」は検出できない (意味)。

### 8.2 構造とドメインの一致

DDD では、コードの構造がドメインの構造と一致することが理想とされる。本ツールは「コードの構造」を可視化する。しかし「ドメインの構造」を知らないため、両者の一致を直接判定できない。

ただし、Ubiquitous Language が IR にメタデータとして注入されれば、「コードの式」と「ドメインの式」の比較ができる可能性がある。これは Phase 6+ の研究テーマ。

### 8.3 美的判断の機械化可能性

「美しいコード = 良いコード」という直観は、形式の整合性が**部分的には**意味の品質を予測することを示唆する。これは認知科学的にも示唆されているが、完全な相関ではない。

本ツールは形式の整合性を測る (美的フィードバック) が、それを「品質保証」とは呼ばない。あくまで**注意誘導のヒューリスティクス**として位置づける。

### 8.4 形式から意味への「示唆」の活用

形式の異常は、意味の問題の**示唆**として機能することが多い:

- 巨大な関数 (形式) → ドメインの責務が曖昧 (意味の示唆)
- 循環依存 (形式) → モジュール境界の設計ミス (意味の示唆)
- 散らばった参照 (形式) → ドメインの集約の不在 (意味の示唆)
- 深さの欠如 (形式) → 抽象化の失敗 (意味の示唆)

これらは「形式が意味を示唆する」境界事例として、本ツールが間接的に意味の問題を浮かび上がらせる経路である。ただし、最終的な意味判断は人間に委ねる。

各 Phase で扱える形式の範囲を整理する。

これは「ツールは形式に徹する」という原則を破るのではなく、**他のチャネルとの協業として意味も扱える**ようになる、という拡張である。

## 10. 哲学的位置づけ: ヴィトゲンシュタインとの対比

本ツールの自己定義は、ヴィトゲンシュタインの **「語りえぬものについては沈黙しなければならない」** (Tractatus 7) という命題と興味深い対比をなす。

**ヴィトゲンシュタイン**: 言語で**明晰に語れる**ことは形式的命題のみ。意味・倫理・美などは「示される」しかなく、語ることはできない。

**本ツール**: コードの**機械的に解析できる**ことは形式的構造のみ。意味・倫理・美などは「示唆される」しかなく、判定することはできない。

両者は驚くほど構造的に似ており、本ツールはヴィトゲンシュタイン的な「形式論理学の徹底」を、ソフトウェア工学の領域で実践していると見ることができる。

「示すことができるが語ることはできない」の含意:
- 形式の整合性が意味の品質を**示唆**する (見せることはできる)
- しかし、ツールはそれを**判定**しない (語ることはできない)
- 判定は人間の領域に残る (示されたものに対する応答)

これは深い哲学的根拠を持つ設計判断であり、本ツールが「真面目に作れば」現代版の Tractatus 的な記号体系として機能する可能性を示唆する。

## 11. 残された問題

### 11.1 形式と意味の境界の曖昧性

「これは形式か意味か」が必ずしも明確でない事例がある。命名規約は形式だが、命名の意味は意味の領域、というように、同じ「命名」でも分解できる。

ツールは境界事例について、判定を保留する (人間に委ねる) ことを既定とする。

### 11.2 文化的に変動する「形式」

ある言語コミュニティでは規範的な形式 (Python の PEP 8) が、別のコミュニティでは異なる。本ツールは「形式の規則」自体を文化的中立にする必要があるが、これは原理的に困難。

解決策: 文化別の設定 (`magia.toml` の `[culture]` セクション) を提供し、規則を切り替えられるようにする。

### 11.3 「形式が意味を示唆する」の自動化

§8.4 で示した「形式から意味への示唆」を、機械的にどの程度まで自動化できるか。完全な自動化は意味の領域に踏み込むことになり、本ツールの原則に反する。

しかし、「示唆の強度」を表示することは形式の領域に留まる。「この形式異常は意味の問題を強く示唆する」「弱く示唆する」のグラデーションを提供することはできる。

これは Phase 5 以降の研究テーマ。

### 11.4 LLM との協業の境界

LLM が意味の評価を行う際、「LLM が何を見て、何を見ていないか」をツールが管理する必要がある。LLM への入力 (コード片、コメント、ドキュメント) は形式の領域、LLM の出力 (評価、提案) は意味の領域。

この境界をユーザーに明示することで、ツールの信頼性を保つ。

## 12. 本文書の位置づけ

本文書 (Appendix F) は、これまでの Appendix A-E が積み上げた**機能と価値**に対する**枠組み**を提供する。

- Appendix A: 何を**描けるか** (式の体系)
- Appendix B: どこまで**スケールするか** (景観)
- Appendix C: なぜ**動くか** (理論的根拠)
- Appendix D: どう**設計支援するか** (Designer Mode)
- Appendix E: どこから**学ぶか** (設計哲学カタログ)
- **Appendix F: 何を扱わないか (射程と限界)** ★本文書

Appendix F は他の Appendix の射程を**限定する**役割を果たす。これがあることで、ツールの自己認識が完成し、過剰な期待や誤った使い方を防ぐことができる。

これは「ツールができないことを言語化することで、できることを明確にする」という、哲学的に重要な作業である。
