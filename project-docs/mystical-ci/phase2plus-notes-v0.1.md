## 1. dev-server 化とリアルタイム可視化 (Phase 2)

### 1.1 静止画から対話への転換

Phase 1 が静止画 SVG を単発生成するのに対し、Phase 2 はエディタの隣に常駐する dev-server を実装する。この移行はツールの性格を根本的に変える。

静止画として PR コメントに貼られるツールは、本質的に**事後の批評**である。フィードバックループは数秒から数分の遅延を持つ。dev-server 化すると、ループが**入力中に閉じる**。`async` を加えた瞬間、対応する補助リングの色が変わる。`try/catch` を消した瞬間、エラー経路の線が天井に抜ける——これは批評ではなく、コードとの**会話**になる。

Vim の `:set hlsearch` や TypeScript の inlay hints が、即時可視化によってコーディング体験そのものを変えたのと同じ階層の体験変容が起きるはずである。

### 1.2 解析パイプラインの3層分離

dev-server 化に伴って、Phase 1 の単純なパイプラインを3層に再設計する必要がある。

**永続キャッシュ層** — ファイル単位のハッシュをキーに AST と IR をディスクキャッシュする。`sled`、`RocksDB`、`sqlite` のいずれかを使用。これは VS Code の TypeScript サーバや rust-analyzer の semantic database と同じ思想で、起動時に全プロジェクトを再解析せず差分だけ計算する。関数単位のハッシュキーを別途持ち、ファイル内で変更された関数だけ IR を再生成することで、数万関数のモノレポでも起動後の応答が数十ミリ秒で済むようにする。

**インクリメンタル再計算層** — `notify` (Rust) または `chokidar` (Node) によるファイル監視 → 差分 AST 抽出 → 影響範囲伝播 → IR 更新 → メトリクス再計算。ここで重要なのは**逆依存グラフ**の維持で、関数 A が関数 B を呼んでいるとき、B の変更が A の図形にも影響することを追跡する必要がある。Salsa のような Rust のインクリメンタル計算フレームワークを使うと、この依存追跡が自然に書ける。

**ストリーミング配信層** — WebSocket でフロントエンドに**差分パッチ**を送る。完成した SVG を毎回送るのではなく、JSON Patch (RFC 6902) のような形式で「リング X の色を変更」「線 Y を追加」というパッチを送り、フロント側で SVG DOM を局所更新する。これにより、エディタで1文字入力するごとに60fps で魔法陣がアニメーションする世界が実現できる。

### 1.3 認知的価値の変化

dev-server 化の本質的な価値は、技術的な高速化ではなく**認知の常駐**にある。これまでのコード可視化ツール (CodeCity, Sourcetrail, CodeScene 等) は、起動して理解するまでに時間がかかり、日常的なコーディング中には使われない。dev-server 化された Mystical は、**周辺視野に常駐する**ツールになる。これは IDE のシンタックスハイライトやエラー下線が常駐していて、認知負荷を増やさずに即座にフィードバックを与えるのと同じ位置に立つ。違いは、構造レベル・アーキテクチャレベルの情報を、美的にフィードバックすることである。

## 2. レイヤーシステム (Phase 2)

### 2.1 関心の軸としてのレイヤー

Phase 1 仕様書の §5 で定義したレイヤーカタログのうち、Phase 1 では `control_flow`, `effects`, `type_info` の3つのみ実装する。Phase 2 ではこれを大幅に拡張し、対話的に切り替え可能にする。

レイヤーは「関心の軸」であり、IR と独立したクエリとしてモデル化される。各レイヤーは IR への問い合わせ関数として記述され、結果を SVG の `<g>` 要素にレンダリングする。Phase 1 仕様書で既に SVG の `<g>` 分離を確保してあるため、Phase 2 では SVG を再生成せず CSS のクラス操作だけで対話的レイヤー切替を実装できる。

### 2.2 レイヤー間の制約

Phase 2 における最も重要な技術的制約は**位置共有**である。同じ関数の同じリングは、どのレイヤー組み合わせでも常に同じ位置に描画されなければならない。これがないと、レイヤーを切り替えるたびに図形が動いてしまい、認知連続性が崩れる。

レイアウトエンジンは「レイヤーに依存しない位置決定」を行い、レイヤーは「既に決まった位置の上に情報を重ねる」という分業を厳密に守る必要がある。Phase 1 の安定レイアウトはこの制約の土台となる。

### 2.3 レイヤーの操作

ユーザーは以下の操作を対話的に行える:

- レイヤーの ON/OFF
- 透明度の調整
- 色相の変更
- 複数レイヤーの AND/OR 合成
- レイヤーセットのプリセット保存・呼び出し

UI は Photoshop のレイヤーパレットを参考にする。

## 3. フィルター言語 (Phase 2)

### 3.1 DSL の必要性

GUI のチェックボックスで全レイヤーを切り替えるだけでも基本機能は成立するが、上級者向けには**クエリ言語**が必要になる。レイヤーの組み合わせ、強調条件、非表示条件を、テキストとして記述・保存・共有できる仕組みである。

これにより、断面の取り方そのものを git でバージョン管理し、チームで共有できるようになる。「セキュリティレビュー断面」「アーキテクチャ監査断面」「新人オンボーディング断面」のようなプリセットが、コーディング規約と並列の**実行可能なチーム知識**として扱える。

### 3.2 構文案

SQL の `SELECT ... WHERE` や jq のフィルタ式に近い思想で、以下のような構文を想定する:

```
show: control_flow + effects[network, db]
highlight: changed_in_pr
hide: test_covered > 0.8
filter: complexity > 7
```

意味:
- 制御フロー層と「ネットワーク/DB の副作用」を表示
- PR で変更された箇所を強調
- テスト被覆80%超は非表示
- 複雑度7超の関数のみ表示

### 3.3 チーム知識の継承装置として

フィルター定義ファイル (例えば `*.mystical` 拡張子) は git で管理される。これはテキスト形式で書かれていた暗黙のレビュー観点が、図形の上に強制力を持って投影される仕組みである。新人が「セキュリティレビューはこの観点で見ます」と教わる代わりに、`security-review.mystical` を読み込めば同じ視点が瞬時に得られる。

## 4. CI 統合 (Phase 3)

### 4.1 PR コメント投稿

GitHub Actions, GitLab CI, Bitbucket Pipelines への統合を実装する。PR ごとに以下を実行する:

1. ベースブランチと PR ブランチの両方を解析
2. 変更された関数だけ図形を生成 (インクリメンタル)
3. 差分画像を PR コメントに自動投稿
4. メトリクス変化をテーブルで併記
5. しきい値超過時は status check を fail

GitHub の Files changed タブの隣に「Spell changed」タブを置くイメージである。

### 4.2 差分エンジン

PR 前後の2つの IR を比較する差分エンジンを Phase 3 で実装する。出力は:

- 構造変化のハイライト (新規リング、削除リング、新規線)
- メトリクス変化のサマリ
- 視覚的 diff (A/B 重ね合わせ、変化箇所のみ強調)

レイヤーごとの差分も提供する。「制御フロー層は変化なし、副作用層に変化あり」のような分解表示。

### 4.3 しきい値判定

CI で fail させる指標は最小限に絞る方針を維持する。Phase 1 仕様書の設計原則「ツールは判決を下さず、注意を誘導する」と整合させるため、大半のメトリクスは「警告マーク」レベルに留め、明確に害がある変化 (例: 循環的複雑度が大幅増加、unsafe ブロックが新規追加、ホットパス上の DB クエリ追加) のみを fail 対象とする。

これは静的解析ツールの過剰判定問題への対処でもある。fail させすぎると無視されるツールになる。

## 5. 多言語アダプタ (Phase 4)

### 5.1 Node.js バックエンド

`mystical-node` アダプタを実装する。TypeScript Compiler API または ts-morph を使用。Node 特有の要素を IR に変換する:

- **ルーティング**: HTTP エンドポイント = エントリポイントの GateGlyph 付き MainRing。`POST /api/users` のような情報をリング外周にラベルとして配置
- **ミドルウェアチェーン**: ルートに到達するまでに通る AuxRing の列
- **DB アクセス**: 召喚記号の中でも特別扱い。テーブル名・クエリ種別 (SELECT/INSERT/UPDATE/DELETE) で記号を変える
- **外部 API 呼び出し**: 別の「外部魔導書」への召喚線
- **DI コンテナ (NestJS)**: 召喚先が抽象シジル → 実装は別の魔導書、というクリーンな構造

Node の動的依存 (動的 require、プロトタイプ拡張) は静的解析で取りきれないため、Phase 5 の動的解析と組み合わせて初めて完全になる。

### 5.2 Web フロントエンド

`mystical-web` アダプタを実装する。React/Vue/Svelte に対応。Web フロント特有の要素:

- **コンポーネントツリー**: コンポーネント1つ = リング1つ。子コンポーネントを内包する補助リング
- **Props/State**: リングの「入力孔」と「内部記号」として区別。Props drilling は深いネストに対する細い線として表現される
- **Hooks**: 特別な召喚記号。`useEffect` の依存配列は線の太さに反映
- **JSX**: 表示要素は補助リングではなく「外周装飾」として描き、ロジックと表示の分離を図形上で明示
- **Context**: 巨大な背景円として描き、参照コンポーネントに細い線
- **状態管理 (Redux/Zustand/Jotai)**: 外部の独立した「召喚円」

ファイル単位ではなく**コンポーネント単位**を主粒度にする点が、Rust や Node と異なる。IR のメタデータに「アダプタごとの粒度宣言」を追加する必要がある。

### 5.3 言語横断比較

Rust のバックエンド関数と Node の同名 API ハンドラを並べて比較したい、というユースケースが想定される (フルスタック開発で「フロントとバックで同じドメイン概念がどう実装されているか」を見たい場合)。

これには IR レベルで「論理的に同じ処理」を正規化する仕組みが必要になる。難易度は高い。Phase 4 後半またはそれ以降で検討する。

## 6. 動的解析と追加レイヤー (Phase 5)

Phase 1〜4 までは静的解析のみだが、Phase 5 では実行時情報の合流を行う。

### 6.1 プロファイル統合

APM (Datadog, New Relic) や OpenTelemetry、Rust の `flamegraph`、Node の `clinic.js` などのプロファイラ出力を IR にフィードバックする。ホットパスを**金色の線**で重ね描く。

これにより「綺麗な魔法陣だが、ここが詰まっている」が初めて見える。静的構造の美しさだけでは決して見えない種類の問題を可視化できる。

### 6.2 テスト被覆統合

カバレッジ情報を魔法陣に重ねる。カバーされていない補助リングは**灰色**にする。これだけで「テストが書かれていない領域」が一目瞭然になり、レビュー時の注意誘導として強力に機能する。

### 6.3 git churn 統合

過去 N 日間の変更行数を関数単位で集計し、温度マップとして重ねる。CodeScene の hotspot 解析の Mystical 版である。

「最近変更頻度が高く、かつテスト被覆が低く、かつ副作用が多い」という複合条件で浮かび上がる関数は、技術的負債のホットスポットの強力な候補となる。

### 6.4 動的依存解決

Rust の `dyn Trait` 経由の動的ディスパッチ、Node の動的 `require()`、TypeScript の `as any` 経由の依存——これらは静的には取りきれない。Phase 5 でテスト実行時のトレーサを仕込み、動的に解決された依存を IR にフィードバックする仕組みを実装する。

動的依存は静的依存とは別レイヤーで描画し、**破線**などで区別する。

### 6.5 AI 注釈チャネル

LLM ベースの解析を別チャネルとして併用する。図形の上に「ここに違和感があるという AI コメント」を吹き出しとして重ねる。図形と AI 注釈の二層構造である。

これにより、構造解析では検出できないドメインスメル (命名の悪さ、ビジネスロジックの誤り、セキュリティ脆弱性の疑い) を補完できる。AI の指摘は判定ではなく注意誘導として位置づける点で、ツール全体の設計原則と整合する。

## 7. 描画様式と解析軸の対応 (Phase 3 + Phase 6)

### 7.1 二大可視化軸の整理

ソフトウェア工学には、コードを可視化する古典的な二大グラフが存在する。

**Call Graph / Dependency Graph** は「誰が誰を呼ぶか」「誰が誰に依存するか」という静的依存関係を表す。AST と直接対応し、Symbol Resolution の範囲で得られる。クラス階層、モジュール構造、関数定義階層も同じ系譜である。

**Data Flow Graph** は「値がどこで生まれ、どこを通り、どこで使われるか」「状態がどう遷移するか」という、データの流れを表す。Reaching Definitions、Use-Def chains、Static Single Assignment、Information Flow、Taint Analysis などの解析が扱う。

Mystical の描画様式 (Drawing Style) は、この二大軸に対応する。

### 7.2 ミッドチルダ式 = Call Graph 可視化

ミッドチルダ式の図形的特徴 (均等な円の多重連環、または内接する正四角形) は、本質的に**ノード中心**の可視化に対応する。中心が明確で、要素が中心からの距離と方位で配置される。Phase 1 仕様書で定義されている「メインリング+補助リング+召喚記号」の同心円構造そのものである。

ミッドチルダ式が向くコードの性格は、**構造化されたモジュラリティ**、**明確な階層**、**整然とした責務分割**。Rust の trait と impl、TypeScript のクラス階層、関数型のコンビネータ合成。要するに、現代的な分割統治が成立しているコード。

「内接する正四角形」バリアントは離散的な4分割を表現するもので、CRUD、4ステートのステートマシン、レイヤードアーキテクチャ (Presentation/Application/Domain/Infrastructure) のような明示的4分割構造に対応する。

必要な解析: AST + Symbol Resolution
実装 Phase: Phase 1 で既に着手

### 7.3 ベルカ式 = Data Flow Graph 可視化

ベルカ式の図形的特徴 (三つの円または三角形を三点に配置し、結んで大きな三角形を作り力場を形成) は、本質的に**エッジ中心**の可視化に対応する。中心がない、あるいは中心が「間」にある。3つのノードそのものではなく、3点が張る三角形の**面**こそが意味を持つ。

ベルカ式が真価を発揮するコード:

- **State + Action + Reducer**: Redux、Elm Architecture、useReducer
- **Model + View + Controller**: MVC, MVVM, MVP
- **Port + Adapter + Domain**: ヘキサゴナルアーキテクチャ
- **Command + Event + State**: イベントソーシング、CQRS
- **Producer + Channel + Consumer**: 並行処理、メッセージング、Actor モデル
- **Test + Spec + Implementation**: TDD/BDD

これらはミッドチルダ式の階層構造では本質が見えなくなる。Reducer の関数本体を巨大な MainRing として描いても、設計の本質である「State と Action と次の State の関係性」が見えない。3点の三角形として描いたとき、初めて構造が意味を持つ。

必要な解析: Data Flow Analysis (Use-Def chains, 関数間解析を含む)
実装 Phase: Phase 3 以降

### 7.4 補完関係と並置の価値

両式は排他的選択ではなく、**同じコードを別の軸で見た補完的なビュー**である。両方を並置すると、「コードは綺麗に書かれている (ミッドチルダ式が整っている) が、データの流れが歪んでいる (ベルカ式の力場が偏っている)」というスメル、あるいはその逆が浮かび上がる。

これは単独のビューでは絶対に検出できない種類の問題である。具体例: リファクタリング不足の Redux コードによくある「reducer は SOLID で綺麗だが、Action の種類が増えすぎて State の遷移が混沌としている」状態。ミッドチルダ式では美しい関数群に見え、ベルカ式では場が破綻して見える。

これは「構造的健全性とデータフロー健全性は別レベルの品質である」という、現代の状態管理で繰り返し学ばれてきた教訓の、視覚化された形である。

### 7.5 自動判定と推奨

AST の形からスタイルを推奨するヒューリスティクスを実装可能。

ベルカ式が推奨される条件:

- 関数シグネチャが `(A, B) -> C` の形で、A と C が同じ型 (Reducer パターン)
- 構造体に明確な3つの責務フィールドがある (MVC 的)
- ディレクトリに `commands/`, `events/`, `states/` が並存する (CQRS)
- 3つのトレイトが相互参照している
- メッセージパッシングのチャネルが3者で形成されている

これらが検出されたとき、ツールは「このコードはミッドチルダ式で描けますが、構造的にはベルカ式が適合します。`--style belka` で再描画しますか?」と提案する。

逆にベルカ式で描こうとして3点関係が明確でない場合、「ベルカ式は3点関係を要しますが、このコードには中心が1つしかありません」とミッドチルダ式へのフォールバックを提案する。

この提案機能はコードレビューの新しい次元を開く。「このコードはどちらの式で描けるべきか」は本質的に「ノード中心の設計か、関係性中心の設計か」という設計レベルの問いに帰着する。

### 7.6 力場のレンダリング

ベルカ式の「力場」は、SVG では3点を頂点とする三角形の内部に**勾配場**を描くことで表現する。各頂点を中心とした放射状グラデーションの重ね合わせ (アディティブブレンド) で実装可能。

場の強度は、3点間のデータフロー量・呼び出し頻度・結合度を反映する。3点のうち1点が過剰に使われていれば、その点を中心とした場が強くなる。これにより、関係性の不均衡——例えば「View だけ肥大化していて、Model と Controller が痩せている MVC」——が**場の歪み**として直感的に見える。

ミッドチルダ式が「ノードの大きさと配置」で情報を伝えるのに対し、ベルカ式は「場の濃淡と歪み」で情報を伝える。

### 7.7 夜天の書式 = メタ循環の可視化

二大軸の整理から導かれる第3式の位置づけは、「コードがデータであるシステム」のグラフ、つまり**コードグラフとデータフローグラフの境界が溶けた領域**の可視化である。

マクロ展開、コード生成、リフレクション、eval、JIT——これらは**コードがデータとして処理される**システムで、ミッドチルダ式とベルカ式の区別自体が成立しない。マクロ展開器に流れ込む「データ」は、展開後にはコードとして再びミッドチルダ式の対象となる。Hofstadter の strange loop、あるいは Curry-Howard 対応における型とプログラムの双対性の、可視化版である。

夜天の書式の魔法陣が「他の魔法陣を内包し、自らを書き換える」のは、メタプログラミングのコードグラフが**自己参照的**であるためである。

実装時には、これは「ミッドチルダ式とベルカ式が同じ図形上で互いに変換し合う、時間方向の動的可視化」になる。Rust のマクロが展開される瞬間、コードの一部がベルカ式の力場に変換され、別の場所でミッドチルダ式のリングとして再凝固する、というアニメーション。技術的に重いが、メタプログラミングの本質を視覚的に伝える唯一の方法になりうる。

必要な解析: マクロ展開トレース、コード生成トレース、リフレクション解析
実装 Phase: Phase 6 以降

### 7.8 三式と Phase の対応表

| 式 | グラフ種別 | 必要な解析 | 主役 | 実装 Phase |
|---|---|---|---|---|
| ミッドチルダ式 | Call Graph / Dependency | AST + Symbol Resolution | Node | Phase 1 |
| ベルカ式 | Data Flow Graph | Dataflow Analysis | Edge / Field | Phase 3 |
| 夜天の書式 | Self-Referential Code/Data | Macro/Codegen Trace | Loop / Transformation | Phase 6+ |

これは美しい階段である。簡単なものから本質的に難しいものへ、扱う情報の抽象度が上がっていく。リリカルなのはのファンタジー設定が、結果として工学的に正しい難易度順序になっていたのは、設定の作り込みの深さの副産物である。

## 8. 立体ビュー (Phase 6)

### 8.1 課題設定の再確認

平面の SVG 1枚で表現できる情報量には認知的な限界がある。複数のレイヤーを同時表示すると視覚的に混雑し、注意誘導という本ツールの目的を損なう。一方、関心レイヤー間の**相関**——例えば「副作用が多く、かつテスト被覆が低く、かつ最近変更頻度が高い領域」のような複合条件——は、複数レイヤーを横断的に見る必要がある。

立体ビューは、各レイヤーを Z 軸上の異なる平面に配置し、視点を回転させることで複数レイヤーの相関を**柱**として浮かび上がらせる。これは平面では原理的に表現できない情報密度を実現する。

### 8.2 Z 軸の使い方の選択肢

**選択肢A: レイヤー積層方式** — 各レイヤーを Z 軸上の異なる平面に配置。手前から「制御フロー」「データフロー」「副作用」「テスト被覆」と積み、視点を回転させると同じリングを貫く柱が見える。建築の重ね図面 (平面図・立面図・電気配線図) を空中で重ね合わせた立体図に近い。

**選択肢B: 抽象度方式** — Z 軸を抽象度の軸として使う。手前が具体的な実装、奥が抽象的なアーキテクチャ層。ズームすると深さが変わる。Google Maps のズームレベルと近い思想で、関数→モジュール→クレート→プロジェクトと、ズームアウトすると魔法陣が大きな魔法陣の一部のリングになるフラクタル構造。

**選択肢C: 時間軸方式** — Z 軸を git の時間軸として使う。手前が main の最新、奥に向かって過去のコミットが積まれる。視点を真横に倒すと、関数の進化が地層として見える。コード考古学的価値があり、技術的負債がどこで発生したか歴史的に追跡できる。

**選択肢D: ハイブリッド** — Z 軸でレイヤー積層、時間軸スライダーを別 UI で持つ。両方を併用すると「3ヶ月前のテスト被覆率と現在の比較」が立体魔法陣のアニメーションとして見える。

推奨は A + D の組み合わせ。Z 軸はレイヤー積層、時間は別の UI スライダー。デフォルト視点 (真上見下ろし) では従来の平面魔法陣に見え、視点を傾けると積層が現れるプログレッシブな体験を提供する。

### 8.3 操作インターフェース

- **クリック**: 関数選択 → 詳細パネル展開
- **ダブルクリック**: 関数の中にズームイン (フラクタル構造のドリルダウン)
- **ホバー**: 接続線の経路追跡 (この関数から伸びる線をすべて発光)
- **ドラッグ**: 視点回転
- **スクロール**: ズーム
- **シフト+スクロール**: レイヤーを1枚ずつ表示/非表示
- **キーボードショートカット**: レイヤーセットのプリセット切替
- **検索バー**: 関数名や記号で検索、対応するリングが発光
- **時間軸スライダー**: git 履歴を遡る

3D の自由視点はマウスでは扱いにくいため、**プリセット角度のスナップ** (上面、側面、斜め45度) を主とし、自由視点はオプションとする。Figma の prototype mode が参考になる。

### 8.4 認知負荷管理

立体表示は人間の認知負荷が高く、常時オンにすると疲労する。デフォルトは平面 (上面視点) にし、必要時のみ立体に切り替える UX が現実的である。VS Code の minimap が「概観したい時だけ見る」位置にあるのと同じ思想。

立体ビューは Phase 1 仕様書の設計原則「複雑な射影は明示的な呼び出しを要求する」を最も強く体現する機能である。明示的なコマンド (例: `mystical invoke-cube` または UI 上の専用ボタン) を経て初めて起動される。

ファンタジー的に立体魔法陣が特殊な状況・権限・代償を要する儀式として描かれるのと同様に、立体ビューは認知的代償と引き換えに通常では到達不可能な情報密度を提供する位置に置かれる。

### 8.5 立体ビューの正当なユースケース

立体ビューが真に必要となるシーンを明確化しておく:

- 四半期に一度のアーキテクチャ全体監査
- 大規模リファクタリング前後の構造比較
- 新規参画者へのコードベース概観の教示
- 異常検知 (立体的な突起の検出)
- カンファレンスや経営報告での視覚資料

これらは頻度は低いが、必要時には平面表現では代替不可能な価値を提供する。「使わない機能」と「存在しない機能」の区別を本ツールが厳密に守る理由は、ここにある。

### 8.6 VR/AR 対応の射程

Phase 6 の発展形として、Apple Vision Pro や Quest 系デバイスでの空間 UI 対応を検討する。コードの周りを歩き回りながらレビューする体験は、空間 UI が成熟しつつある今、現実的な選択肢となりつつある。Mystical の魔導書美学は、VR コードレビューの**最も気高いインターフェース言語**になりうる。

## 9. アクセシビリティ

### 9.1 呪文書き起こし

図形ベース表現の視覚障害者対応として、同じ IR から構造化テキストの「呪文書き起こし」を生成する機能を Phase 2 以降で実装する。スクリーンリーダーで読める、メトリクスベースのテキスト要約。

例:
```
関数 calculate_total:
  メインリングは中規模 (12操作)。
  3つの補助リングを持つ。
  外部呼び出し: db::query (DB副作用、1回)、logger::info (IO副作用、2回)。
  早期リターンが2経路。Result型を返す。
  Phase 1 メトリクス: 複雑度4、副作用カテゴリ2種。
```

### 9.2 段階的実装

Phase 2 で基本的な書き起こし機能、Phase 3 でレイヤー切替に対応した書き起こし、Phase 5 で動的解析情報を含む包括的書き起こしへと拡張する。

## 10. 記号語彙の拡張 (Phase 4+)

### 10.1 ドメイン特化記号集

Mystical 原典の記号セットは基本演算しかカバーしていない。CI 用途では、HTTP、gRPC、DB (SQL/NoSQL別)、ファイル IO、暗号化、認証、ログ出力、メトリクス送信、キュー、キャッシュなどの**ドメイン演算の記号体系**を整備する必要がある。

### 10.2 コミュニティ拡張モデル

記号集は標準ライブラリではなく**拡張シジル集**として、コミュニティで継続的に育てる仕組みを想定する。

- `mystical-glyph-aws`: AWS SDK 用記号
- `mystical-glyph-gcp`: GCP SDK 用記号
- `mystical-glyph-grpc`: gRPC 用記号
- `mystical-glyph-kafka`: Kafka 用記号

クレートとして配布し、`Cargo.toml` で参照する形にする。

## 11. 思想的整理

### 11.1 設計原則の再確認

Phase 1 仕様書 §2 で定めた5原則は Phase 2 以降も貫かれる:

1. 完全な情報を内部に保持する
2. デフォルトは最も単純な射影を提示する
3. 射影は明示的に拡張・縮小できる
4. 複雑な射影は明示的な呼び出しを要求する
5. 任意の射影は保存・共有・バージョン管理できる

dev-server 化、レイヤーシステム、立体ビュー、3式の選択、フィルター言語——これら全てが上記5原則の異なる側面の実装である。

### 11.2 能力とデフォルトの分離

「読み取りが困難 (だからデフォルトでは表示しない)」と「機能として存在しない」を厳密に区別する設計を維持する。立体ビューを年に一度しか使わないユーザーが大半でも、必要なときに必ず使える状態であることが、ツールの**到達天井**を決める。

これは Larry Wall の "make the easy things easy and hard things possible" の系譜である。Git、Vim、Photoshop、Emacs と同じ階層のツール設計原則。

### 11.3 ファンタジー的メタファーの正しい解釈

「立体魔法陣はファンタジーにおいてさえ異質な能力である」という観察は、ツール設計の指針として有用である。立体ビューはツールの天井を決める機能であり、デフォルトでは隠れていて、特定の操作・認証・状況を経て呼び出される。物語が「強い力には代償が伴う」と教えるとき、それは認知科学的に「強い表現は強い負荷を伴う」という現実を反映している。

3式の名称をリリカルなのは由来とすることに技術的な意味はないが、設定が緻密に作り込まれたファンタジー宇宙は、ツール命名と分類のメタファー集として高い解像度を持つ。実装時には中立的な訳語 (例: 近代式 / 古典式 / 自己参照式) も併記する形が現実的である。

## 12. ロードマップ再整理

Phase 1 仕様書のロードマップを、本文書の議論を踏まえて補強する:

**Phase 1**: 単一 Rust 関数を SVG 1枚の平面魔法陣として出力する CLI。ミッドチルダ式 (ConcentricRings) のみ。

**Phase 3**: CI 統合 (PR コメント、差分エンジン、しきい値判定)、データフロー解析の導入とベルカ式描画の有効化。

**Phase 4**: 他言語アダプタ (Node.js, Web フロント)、記号語彙の拡張モデル整備、言語横断比較の検討。
