# まぎあマギカ計画 — コンセプト文書

> コードベースを「魔法陣の積層」として立体表現する可視化システムの構想
> 作成日: 2026-06-10（モバイルでの構想セッションより）

---

## 1. コンセプト

プログラムの構造を、ファンタジー世界の魔法陣として空間にマッピングする。単なるビジュアル趣味ではなく、**空間認知（近接・グルーピング・境界・対立）を母語とする思考者のためにコードを翻訳する装置**である。フローチャートより魔法陣のほうが「読める」人のためのIDE、あるいはコードレビュー支援ツール。

着想の起点は魔法陣型プログラミング言語「**Mystical**」のGIGAZINE記事（2025-05-19, https://gigazine.net/news/20250519-mystical/ ）。Mysticalは文字ではなく図形を命令として用い、リング（内外に境界を持つ円形の帯）構造に基づき、最外周リングの3時の位置から反時計回りに実行される。実装はPostScriptベース（公式: https://suberic.net/~dmm/projects/mystical/ ／リポジトリ: https://github.com/denismm/mystical_ps ）。

Mysticalが「魔法陣でコードを書く」言語であるのに対し、本計画はその逆方向、**既存のコードを魔法陣として読む**アプローチを取る。

## 2. 基本メタファー

### モジュール = 魔法陣

各モジュール（パッケージ／クラス／ファイルなど、粒度は設定可能）をひとつの魔法陣として描画する。

- **内部の紋様の緻密さ** = モジュール内部のロジックの豊かさ
- **外周の接点（ルーンの突起）** = 公開インターフェース

これにより POSD の「深いモジュール」がそのまま視覚化される。

- **深いモジュール**: 内側は緻密で濃いのに、外周の接点はほんの数カ所 → 美しい陣
- **浅いモジュール**: 中身スカスカなのに接点だらけの貧相な円
- **パススルー関数**: ただ信号を中継するだけの空っぽの輪として一目でバレる

### 参照 = 魔力の流れ

依存関係はルーン同士を結ぶ光の線として描く。健全な依存は一方向に流れる。

- **循環参照**: 光が回り続けて収束しないウロボロス状の禍々しい模様として現れる。「なんかこの陣、禍々しいな」がそのまま静的解析になる

## 3. パラダイム = 流派

プログラミングパラダイムによって陣の「作風」が変わる。同じ仕様でも流派で見た目が変わる。

- **関数型**: DAG構造なので、魔力が上から下へ滝のように流れる整然とした陣
- **オブジェクト指向**: 相互参照の網目による曼荼羅
- **レイヤードアーキテクチャ**: 同心円の結界構造

頻出パターンにはパラダイム用語ではなくファンタジー語彙で命名できる（次節）。

## 4. ファンタジー命名辞書（初版）

命名がそのまま構造のドキュメントになることを狙う。

| コード上の概念 | 魔法陣としての名前 | 含意 |
|---|---|---|
| Singleton | 召喚陣 | 世界に一体しか顕現できない契約 |
| Observer | 使い魔契約陣 | 主から従への一方的な感知。従の増減は主に影響しない |
| イベントバス | 共鳴陣 | 直接繋がっていないのに振動が伝わる |
| グローバル状態への依存 | 世界樹直結 | 便利だが世界樹が枯れたら全員死ぬ |
| 神クラス | 禁呪の大儀式陣 | 強大だが誰も全容を理解しておらず、解体しようとした者から犠牲になる |
| 純粋関数 | 等長紋章 | ノルム保存・副作用なし・コストなし |
| 循環参照 / 無限再帰 | 自己参照紋章（ゲーデル崩壊） | 最も忌避される魔法事故。スタックオーバーフローの完璧なメタファー |

※「等長紋章」「ゲーデル崩壊」は既存の自作魔法体系（2026-04「回路図形による魔法体系の構想」セッション）からの流用。同セッションは本計画の用語集として再利用価値が高い。

## 5. 空間配置の理論 — 2D隣接と3D積層

### 2D: 隣接性が結合を語る

陣は2次元平面上にマッピングされるため「隣接」と「遠方」が生まれる。

- 良い設計では、ほとんどの参照が**隣接する陣**との間で完結する（参照の局所性）
- 遠方の陣への参照は、隣接という契約を飛び越える「越境」として可視化される
- 良い設計は平面に置いただけで線がほぼ交差しない

### 3D: 積層しても美しいか

レイヤーごとの平面を縦に積層し、魔法塔として表現する。

- **美しい状態**: 層をまたぐ参照が真下の接点にまっすぐ落ちる。各階の陣を縦に貫く光の柱が揃っている
- **醜い状態**: 3階の陣から7階の陣へ斜めに突き刺さる越境の光線。層を無視して絡まる蜘蛛の巣。らせん状に登って自分に戻る循環参照
- 世界法則: **「魔力は重力に従って下に流れるべし」** = レイヤードアーキテクチャの「上位層は下位層にのみ依存せよ」

アーキテクチャ違反が「禍々しさ」という審美的直感として即座に検知できる。

## 6. 美しさ = メトリクス（定量化）

メタファーを超えて、実際に計測可能な品質スコアにする。

- グラフ描画理論における**辺の交差数**と**総辺長**の最小化は、そのまま結合度の指標
- 力学モデル（force-directed layout）で3D配置を最適化した後に**残ってしまった**長い辺・交差・層貫通の数 = 定量的なコード品質スコア
- つまり「最適配置してもなお醜い」コードは構造的に醜い、と数値で言える

## 7. 技術方向性（仮・要検討）

- **解析**: 対象言語のAST解析 → モジュール境界と依存グラフの抽出（言語ごとにアダプタ）
- **配置**: 力学モデルによる2Dレイアウト + レイヤー検出による積層
- **描画**: Three.js / WebGL による3D表示。陣の紋様はモジュールのメトリクス（複雑度・行数・公開API数）から手続き的に生成
- 詳細な技術選定は別セッションで実施予定
